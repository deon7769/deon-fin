## 1. Contexto & objetivo

Hoje o frontend é o **legado server-rendered** em `src/web/` (`templates/index.html` 377 linhas + `static/app.js` 1010 + `static/app.css` 214), com Chart.js + marked.js e tema **roxo** (`--accent: #7c5cff`). Esse front continua funcionando e **não pode ser quebrado** (primer §2.2). A nova UI é um app **Next.js** independente, construído **em paralelo**, na pasta **`web/`** na raiz do repositório (primer §3.1), com o tema **escuro de acento amarelo** do print (primer §7).

A F0.1 já entregou (ou entrega antes desta) a **fundação de API**: base `/api`, envelope de erro `{ error: { code, message } }`, CORS liberado para `http://localhost:3000`, auth Basic desligada quando `APP_PASSWORD` vazio (dev). O frontend desta spec consome essa API.

**Objetivo da F0.2:** deixar o projeto Next pronto para que F0.3 e F2.* implementem telas **sem reconfigurar nada de base**. Concretamente:

1. Inicializar **Next.js 14+ (App Router) + TypeScript + Tailwind** em `web/`, com a árvore de pastas, scripts npm, `.env` e `next.config` (preparado para `basePath`/proxy — primer §3.2).
2. Materializar o **design system** do primer §7 no `tailwind.config` (tokens de cor, radius, fonte Inter). Escuro como default; **ganchos** para o tema claro (a *alternância* é F0.3).
3. Montar o **shell de layout**: `<Sidebar>` 240px (grupos `MENU`/`OUTROS` com as rotas do §7, item ativo amarelo, ícones `lucide-react`) + `<Header>` (slots para `MonthYearPicker` e olho de privacidade — **placeholders** aqui; reais em F0.3) + rodapé da sidebar (`Tema`/`Ocultar`, placeholders).
4. Configurar o **provider do TanStack Query** e o cliente HTTP `web/lib/api.ts` (baseURL via env, parse do envelope de erro de F0.1, credenciais conforme auth).
5. Criar os **componentes-base como stubs** com props tipadas e uma **tabela de PROPS** por componente: `KpiCard`, `SectionCard`, `DataTable`, `Pill`, `ProgressBar`, `MoneyText` (interface já pronta para o "Ocultar" de F0.3), `EmptyState`, `BucketSelect`/`TagSelect` (só casca/props).
6. Criar os **utils de formatação** `web/lib/format.ts` (`formatBRL`, `formatDate`, `formatPercent`, `formatMonthYear`, pt-BR) com exemplos.
7. Criar as **rotas placeholder** de todas as telas do menu (`/`, `/orcamento`, `/metas`, `/contas`, `/faturas`, `/transacoes`, `/tags`, `/perfil`, `/faq`).

> **Princípio "stub vs. finalizado":** F0.2 entrega **estrutura e contratos** — props tipadas, layout visual, fiação de dados. **Não** entrega lógica de feature (nenhuma chamada real a `/api/painel`, nenhum cálculo de KPI, nenhum estado global de período/Ocultar/Tema). Onde a lógica não existe, o componente renderiza um placeholder visual coerente (skeleton/`EmptyState`/`—`), **nunca dados falsos**. Cada bloco abaixo marca explicitamente o que é **stub** e o que é **finalizado** nesta spec.

## 2. Escopo

### 2.1 Dentro do escopo (F0.2 entrega)

- **Projeto Next** em `web/`: `package.json` (deps + scripts), `tsconfig.json`, `next.config.mjs`, `postcss.config.mjs`, `tailwind.config.ts`, `.env.local`/`.env.example`, `.eslintrc`, `app/globals.css`, fonte Inter.

- **Provider finalizado:** `<Providers>` com `QueryClientProvider` (TanStack Query) + `QueryClient` configurado.
- **Cliente HTTP finalizado:** `web/lib/api.ts` (`apiFetch`/`api`), baseURL via `NEXT_PUBLIC_API_URL`, parse do envelope `{ error: { code, message } }` → `ApiError`, `credentials: 'include'`.
- **Utils de formatação finalizados:** `web/lib/format.ts` (`formatBRL`, `formatDate`, `formatPercent`, `formatMonthYear`).
- **Componentes-base como stubs (props tipadas, visual mínimo):** `KpiCard`, `SectionCard`, `DataTable<T>`, `Pill`, `ProgressBar`, `MoneyText`, `EmptyState`, `BucketSelect`, `TagSelect`.
- **Rotas placeholder:** uma `page.tsx` por rota do menu, cada uma renderizando `<SectionCard>` + `<EmptyState>` com o nome da tela e o spec que a implementa.
- **Tipos compartilhados** (`web/lib/types.ts`): contratos mínimos (`Bucket`, `Tag`, `Page<T>`, `ApiErrorShape`) alinhados ao primer §4/§6 — só o necessário para tipar os stubs.
- **Documentação de execução:** como rodar dev (Next :3000 + API :8000) e nota de build/prod.

### 2.2 Fora do escopo (pertence a outras specs)

- **Estado global** de período (`MonthYearPicker` real), privacidade (`Ocultar`) e **tema (toggle claro/escuro)** — tudo **F0.3**. Aqui há apenas **placeholders** e os **ganchos** (CSS variables, prop `hidden` em `MoneyText`, slot no `<Header>` e botões no rodapé da sidebar sem `onClick` funcional).
- **Telas de feature** (Painel/KPIs reais, tabela de transações com filtros, orçamento, faturas, contas, metas, perfil) — **F2.***. As `page.tsx` desta spec são placeholders.
- **Chamadas reais à API de negócio** (`/api/painel`, `/api/transactions`, `/api/buckets`, `/api/tags`…). O `api.ts` é o **cliente** pronto; quem o usa para buscar dados são as features. (Exceção opcional de fumaça: F0.2 pode validar `GET /api/health` num ponto de diagnóstico — ver §12.)
- **Lógica dos selects** `BucketSelect`/`TagSelect` (buscar opções de `/api/buckets`/`/api/tags`, persistir via `PATCH /api/transactions/{id}`) — **F1.1/F1.2/F2.2**. Aqui só **casca + props**.
- **Gráficos (Recharts)** — entram quando o Painel/Orçamento forem implementados (F2.1/F2.3). F0.2 só declara a dependência (instalável) para não retrabalhar o `package.json`.
- **Autenticação de produção** do front (reverse proxy / Route Handlers com header Basic) — decidida no spec de deploy; F0.2 deixa `credentials: 'include'` e o `next.config` prontos (primer §3.2, F0.1 §5.4).
- **Descontinuar `src/web/`** legado — não se toca; nova UI vive em `web/`.

> **`NEXT_PUBLIC_` é obrigatório** para a variável ser exposta ao bundle do browser (regra do Next). Como o cliente HTTP roda no navegador (componentes client), a baseURL precisa desse prefixo. Em produção, se o front for servido na **mesma origem** da API (primer §3.2), basta `NEXT_PUBLIC_API_URL=/api`.

**`next.config.mjs`** — pronto para `basePath`/proxy (primer §3.2), sem ativá-los agora:

```js
// web/next.config.mjs
/** @type {import('next').NextConfig} */
const nextConfig = {
  reactStrictMode: true,

  // PRONTO p/ prod (primer §3.2) — ativado no spec de deploy, não aqui:
  // basePath: process.env.NEXT_BASE_PATH || undefined,   // se servir sob /app
  // async rewrites() {                                    // proxy same-origin p/ a API
  //   const api = process.env.API_PROXY_TARGET;           // ex.: http://127.0.0.1:8000
  //   return api ? [{ source: '/api/:path*', destination: `${api}/api/:path*` }] : [];
  // },
};
export default nextConfig;
```

> **Dev não usa rewrites:** em desenvolvimento o front chama `NEXT_PUBLIC_API_URL` (cross-origin para `127.0.0.1:8000`) e o CORS de F0.1 (§5.3) libera `localhost:3000`. O `rewrites` comentado é a **opção same-origin de produção** (evita CORS e permite o reverse proxy injetar Basic Auth — F0.1 §5.4, estratégia 1/2). Deixar comentado cumpre "deixar o `next.config` pronto" sem alterar o comportamento de dev.

**`providers/Providers.tsx`** — TanStack Query (finalizado). É o ponto onde F0.3 vai pendurar os contextos de período/Ocultar/Tema (ganchos comentados).

```tsx
// web/providers/Providers.tsx
"use client";
import { QueryClient, QueryClientProvider } from "@tanstack/react-query";
import { useState } from "react";

export function Providers({ children }: { children: React.ReactNode }) {
  // useState garante 1 QueryClient por montagem (evita compartilhar cache entre requests no SSR).
  const [client] = useState(
    () =>
      new QueryClient({
        defaultOptions: {
          queries: {
            staleTime: 30_000,          // 30s: evita refetch agressivo
            retry: 1,
            refetchOnWindowFocus: false,
          },
        },
      }),
  );

  // F0.3 envolverá children com PeriodProvider / PrivacyProvider / ThemeProvider AQUI.
  return <QueryClientProvider client={client}>{children}</QueryClientProvider>;
}
```

#### `DataTable<T>` ✅ (genérico, com scroll-x e estados loading/vazio)

Tabela genérica tipada por coluna. Em F0.2 é totalmente funcional como **apresentação** (renderiza linhas a partir de `rows`+`columns`); ordenação/paginação/seleção ficam para as telas (F2.*), que passam dados já paginados pela API.

```tsx
// web/components/ui/DataTable.tsx
import { cn } from "@/lib/cn";
import { Skeleton } from "./Skeleton";
import { EmptyState } from "./EmptyState";

export type Column<T> = {
  key: string;
  header: string;
  /** alinhamento; números à direita por convenção (primer/legado .num) */
  align?: "left" | "right" | "center";
  /** render custom; default = String(row[key as keyof T]) */
  cell?: (row: T) => React.ReactNode;
  className?: string;
};

export type DataTableProps<T> = {
  columns: Column<T>[];
  rows: T[];
  rowKey: (row: T) => string | number;
  loading?: boolean;
  /** mensagem/CTA quando vazio (EmptyState) */
  emptyTitle?: string;
  emptyDescription?: string;
  className?: string;
};

export function DataTable<T>({
  columns, rows, rowKey, loading, emptyTitle = "Nada por aqui", emptyDescription, className,
}: DataTableProps<T>) {
  if (loading) return <Skeleton className="h-40 w-full" />;
  if (rows.length === 0) return <EmptyState title={emptyTitle} description={emptyDescription} />;

  const alignCls = (a?: Column<T>["align"]) =>
    a === "right" ? "text-right" : a === "center" ? "text-center" : "text-left";

  return (
    <div className={cn("overflow-x-auto", className)}>
      <table className="w-full border-collapse text-sm">
        <thead>
          <tr className="border-b border-border">
            {columns.map((c) => (
              <th key={c.key} className={cn("px-3 py-2 font-medium text-muted", alignCls(c.align))}>
                {c.header}
              </th>
            ))}
          </tr>
        </thead>
        <tbody>
          {rows.map((row) => (
            <tr key={rowKey(row)} className="border-b border-border/60 hover:bg-surface2">
              {columns.map((c) => (
                <td key={c.key} className={cn("px-3 py-2 text-text", alignCls(c.align), c.className)}>
                  {c.cell ? c.cell(row) : String((row as Record<string, unknown>)[c.key] ?? "")}
                </td>
              ))}
            </tr>
          ))}
        </tbody>
      </table>
    </div>
  );
}
```

#### `ProgressBar` ✅

Barra de progresso (gasto vs. previsto — base de Orçamento F2.3). Estado "excedido" (>100%) usa `negative`; opcional `marker` (meta) como no legado.

```tsx
// web/components/ui/ProgressBar.tsx
import { cn } from "@/lib/cn";

export type ProgressBarProps = {
  /** 0..100 (já em %). Acima de 100 satura visualmente e marca como excedido. */
  value: number;
  /** marca opcional (ex.: 100% = meta) em % */
  marker?: number;
  tone?: "accent" | "positive" | "auto";    // 'auto' = positive até 100, negative acima
  className?: string;
};

export function ProgressBar({ value, marker, tone = "auto", className }: ProgressBarProps) {
  const pct = Math.max(0, Math.min(100, value));
  const exceeded = value > 100;
  const fill =
    tone === "auto" ? (exceeded ? "bg-negative" : "bg-positive")
    : tone === "positive" ? "bg-positive" : "bg-accent";
  return (
    <div className={cn("relative h-3 w-full overflow-hidden rounded-pill border border-border bg-bg", className)}>
      <div className={cn("h-full rounded-pill transition-[width] duration-500", fill)} style={{ width: `${pct}%` }} />
      {marker !== undefined && (
        <div className="absolute inset-y-[-2px] w-0.5 bg-text/60" style={{ left: `${Math.min(100, marker)}%` }} aria-hidden />
      )}
    </div>
  );
}
```

### 3.9 Cliente HTTP — `web/lib/api.ts` (contrato)

Cliente fino sobre `fetch`. Responsabilidades: (1) montar a URL a partir de `NEXT_PUBLIC_API_URL`; (2) enviar/parsear JSON; (3) **enviar credenciais** (`credentials: "include"`, casado com `allow_credentials=True` do CORS de F0.1 §5.3, para o Basic Auth de produção); (4) **parsear o envelope de erro** de F0.1 (`{ error: { code, message } }`) e lançar um `ApiError` tipado.

```ts
// web/lib/api.ts
import type { ApiErrorShape } from "./types";

const BASE_URL = (process.env.NEXT_PUBLIC_API_URL ?? "http://127.0.0.1:8000/api").replace(/\/$/, "");

/** Erro tipado a partir do envelope de F0.1: { error: { code, message } }. */
export class ApiError extends Error {
  constructor(
    public readonly code: string,
    message: string,
    public readonly status: number,
  ) {
    super(message);
    this.name = "ApiError";
  }
}

type Query = Record<string, string | number | boolean | null | undefined>;

function buildUrl(path: string, query?: Query): string {
  const url = new URL(`${BASE_URL}${path.startsWith("/") ? path : `/${path}`}`);
  if (query) {
    for (const [k, v] of Object.entries(query)) {
      if (v !== undefined && v !== null) url.searchParams.set(k, String(v));
    }
  }
  return url.toString();
}

async function apiFetch<T>(
  path: string,
  opts: { method?: string; query?: Query; body?: unknown; signal?: AbortSignal } = {},
): Promise<T> {
  const { method = "GET", query, body, signal } = opts;
  const res = await fetch(buildUrl(path, query), {
    method,
    signal,
    credentials: "include",                 // envia Basic Auth em prod (F0.1 §5.4)
    headers: body !== undefined ? { "Content-Type": "application/json" } : undefined,
    body: body !== undefined ? JSON.stringify(body) : undefined,
  });

  // 204/empty → sem corpo
  const text = await res.text();
  const data = text ? JSON.parse(text) : null;

  if (!res.ok) {
    // Envelope de F0.1: { error: { code, message } }. Fallback se vier outro shape.
    const env = data as ApiErrorShape | null;
    const code = env?.error?.code ?? "http_error";
    const message = env?.error?.message ?? `Erro ${res.status}`;
    throw new ApiError(code, message, res.status);
  }
  return data as T;
}

export const api = {
  get:   <T>(path: string, query?: Query, signal?: AbortSignal) => apiFetch<T>(path, { query, signal }),
  post:  <T>(path: string, body?: unknown) => apiFetch<T>(path, { method: "POST", body }),
  patch: <T>(path: string, body?: unknown) => apiFetch<T>(path, { method: "PATCH", body }),
  del:   <T>(path: string) => apiFetch<T>(path, { method: "DELETE" }),
};
```

### 3.10 Utils de formatação — `web/lib/format.ts`

Formatação **pt-BR** via `Intl` (primer §6: moeda no front; datas `dd/MM/yyyy`). Sem dependências externas.

```ts
// web/lib/format.ts

const BRL = new Intl.NumberFormat("pt-BR", { style: "currency", currency: "BRL" });
const PCT = new Intl.NumberFormat("pt-BR", { style: "percent", maximumFractionDigits: 1 });
const DATE = new Intl.DateTimeFormat("pt-BR", { day: "2-digit", month: "2-digit", year: "numeric" });
const MONTH_YEAR = new Intl.DateTimeFormat("pt-BR", { month: "long", year: "numeric" });

/** R$ 1.234,56 — primer §6 (Intl pt-BR, currency BRL). */
export function formatBRL(value: number): string {
  return BRL.format(Number.isFinite(value) ? value : 0);
}

/** dd/MM/yyyy a partir de Date ou ISO 'YYYY-MM-DD' (primer §6). */
export function formatDate(input: Date | string): string {
  const d = typeof input === "string" ? new Date(`${input}T00:00:00`) : input;
  return DATE.format(d);
}

/** Percentual pt-BR. Por padrão recebe FRAÇÃO (0.5 → "50%").
 *  Passe asFraction=false se já tiver o número em pontos percentuais (50 → "50%"). */
export function formatPercent(value: number, asFraction = true): string {
  return PCT.format(asFraction ? value : value / 100);
}

/** "junho de 2026" a partir de 'YYYY-MM' (mês de competência — primer §5.4). */
export function formatMonthYear(ym: string): string {
  const [y, m] = ym.split("-").map(Number);
  return MONTH_YEAR.format(new Date(y, (m ?? 1) - 1, 1));
}
```

**Exemplos (viram casos de teste — §11):**

| chamada | resultado |
|---|---|
| `formatBRL(1234.5)` | `"R$ 1.234,50"` |
| `formatBRL(-80)` | `"-R$ 80,00"` |
| `formatBRL(0)` | `"R$ 0,00"` |
| `formatDate("2026-06-20")` | `"20/06/2026"` |
| `formatDate(new Date(2026,0,5))` | `"05/01/2026"` |
| `formatPercent(0.5)` | `"50%"` |
| `formatPercent(12.5, false)` | `"12,5%"` |
| `formatMonthYear("2026-06")` | `"junho de 2026"` |
| `formatMonthYear("2026-12")` | `"dezembro de 2026"` |

> **Nota de fuso (data):** `formatDate` ancora strings ISO em `T00:00:00` **local** para evitar o off-by-one clássico (uma data "2026-06-20" interpretada como UTC pode exibir 19/06 em fusos negativos como o do Brasil). Datas vêm da API como `YYYY-MM-DD` (primer §6), então essa ancoragem é suficiente.

## 4. Integração com a API F0.1 (CORS / erros / auth dev)

- **Base URL:** `NEXT_PUBLIC_API_URL=http://127.0.0.1:8000/api` em dev (primer §3.2). O `api.ts` (§3.9) remove barra final e monta `${BASE}/path`.
- **CORS (dev):** F0.1 §5.3 libera `http://localhost:3000` com `allow_credentials=True`. O cliente usa `credentials: "include"`; em dev (sem `APP_PASSWORD`) isso é inócuo, em prod carrega o Basic Auth. **Atenção operacional:** Next dev serve em `localhost:3000`; garanta que `CORS_ORIGINS` do backend inclua exatamente essa origem (e, se você abrir via `127.0.0.1:3000`, adicione-a também — origens são casadas literalmente).
- **Envelope de erro:** F0.1 §5.2 padroniza `{ error: { code, message } }` com status HTTP adequado (`400/401/403/404/409/422/500/502`). O `apiFetch` lê esse envelope e lança `ApiError{ code, message, status }`. As telas (F2.*) tratam `ApiError` (toast/retry/EmptyState de erro); F0.2 só fornece o mecanismo.
- **Auth dev:** F0.1 §5.4 — `APP_PASSWORD` vazio ⇒ sem auth. Logo, em dev o front chama a API livremente. **Auth prod** (Basic via reverse proxy ou Route Handler com header a partir de env do servidor) é decidida no spec de deploy; o `credentials: "include"` e o `next.config` (rewrites comentado, same-origin) já deixam os dois caminhos abertos (primer §3.2; F0.1 §5.4 estratégias 1–3).
- **Compat:** o front legado em `src/web/` (servido por `/` no backend, F0.1 §7) **não** é tocado. A nova UI vive em `web/` e é processo separado (`next dev`). Não há conflito de porta (3000 ≠ 8000).
- **Smoke opcional de conectividade:** F0.2 pode validar a fiação com um `api.get("/health")` num componente de diagnóstico ou teste manual (ver §12), sem acoplar nenhuma tela a dados de negócio.

## 5. Acessibilidade & responsividade

**Acessibilidade (primer §7 "Acessível: foco visível, labels, contraste AA"):**
- **Foco visível** global em `:focus-visible` (outline amarelo `accent`, §3.5).
- **Navegação semântica:** `<aside>`/`<nav>`/`<main>`/`<header>`; link ativo com `aria-current="page"` (Sidebar §3.6).
- **Labels:** selects (`BucketSelect`/`TagSelect`) têm `aria-label`; botões-placeholder (Tema/Ocultar/olho) têm `aria-label`/`title` indicando "em breve / F0.3"; ícones decorativos com `aria-hidden`.
- **Contraste AA:** tokens escuros do primer §7 atendem AA para texto sobre superfícies. O **amarelo `accent` (#F5B301) como texto sobre fundo escuro passa AA**; sobre fundo claro, o gancho `.light` usa `#C98A00` (mais escuro) — F0.3 valida no toggle.

**Responsividade (primer §7 "sidebar colapsável, tabela com scroll-x"):**
- **Sidebar:** em F0.2 é fixa 240px no desktop. O **colapso em telas estreitas** (drawer/hambúrguer) fica para **F0.3** (que já vai mexer no shell para o estado global); F0.2 deixa o layout com `flex` que não quebra (conteúdo ocupa o resto). *Decisão registrada:* não implementar o drawer aqui evita retrabalho quando F0.3 introduzir o estado de UI; documentado como gancho.

- **Conteúdo:** `<main>` com `overflow-x-hidden` e paddings responsivos (`p-6`); cards usam grid fluido nas telas (definido por cada feature em F2.*).
