## 1. Contexto & objetivo

O backend atual (`src/web/app.py` → `create_app()`) é um app FastAPI monolítico: todos os
endpoints estão definidos dentro da factory, e o acesso a dados é feito ora pelos métodos de
`Database` (`src/storage/db.py`), ora por agregações inline (`/api/summary` itera `db.list_transactions`
e soma à mão; `/api/dashboard` chama `build_financial_context`). O schema do SQLite é criado por
`self._conn.executescript(SCHEMA)` no `Database.__init__` (`db.py:112`), com `CREATE TABLE IF NOT EXISTS`.
**Não existe runner de migrations** — o primer (§4) já registra isso e decide que F0.1 cria um.

A nova UI (Next.js) precisa de campos e tabelas que **ainda não existem** (potes/`budget_buckets`,
`tags`, `bucket_rules`, `profile`, `account_balances`, e 6 colunas novas em `transactions`), de uma
API REST organizada por domínio sob `/api`, de um contrato de erro consistente, de CORS para o
front em `http://localhost:3000`, e de utilitários canônicos (`reference_month`, helpers de sinal).

**Objetivo da F0.1:** preparar essa fundação **sem implementar telas e sem quebrar nada do que já roda
em produção** (front legado em `/`, endpoints `/api/summary`, `/api/dashboard`, `/api/cartao`, sync Pluggy,
dedup por fingerprint). Concretamente:

1. Criar `src/storage/migrations.py` — lista ordenada de migrations idempotentes + `apply_migrations(conn)`,
   chamado dentro de `Database.__init__` logo após `executescript(SCHEMA)`.
2. Materializar **todo o schema novo** do primer §4.2 e §4.3 (estruturas vazias + índices; **sem seeds** —
   os seeds dos 6 potes e das 7 tags são de F1.1/F1.2).
3. Introduzir uma **camada de repositórios** (`src/web/repositories/`) para isolar SQL dos routers.
4. Quebrar a API em **routers por domínio** montados sob `/api`, com **handler global de erros**.
5. **CORS** + estratégia de **auth dev/prod**.
6. **Utils canônicos**: `reference_month(posted_at, start_day)` e a materialização/recálculo de
   `transactions.reference_month`; promover os helpers de sinal para uso compartilhado.
7. **Backfill** único de `reference_month` nas transações existentes.

> **Nota de leitura factual (divergência importante — ver §10.4):** os helpers de sinal
> `spending_value`/`income_value` citados pelo primer §5.3 **NÃO existem no código hoje**. Eles foram

## 3. Migrations

### 3.1 Estratégia (idempotente, ordenada, não destrutiva)

Princípios (alinhados ao primer §4):

1. **Nunca destrutivo.** Só `CREATE TABLE IF NOT EXISTS`, `CREATE INDEX IF NOT EXISTS` e
   `ALTER TABLE ... ADD COLUMN` **guardado** por checagem em `PRAGMA table_info(...)`. Nada de `DROP`,
   `DELETE`, nem `ALTER ... RENAME/DROP COLUMN` (SQLite só ganhou `DROP COLUMN` no 3.35, e mesmo assim
   não usamos).
2. **Idempotente.** Rodar `apply_migrations` 2× (ou N×) não falha e não duplica nada. Isso é garantido
   por (a) os guards de existência e (b) uma tabela de controle `schema_migrations` que registra quais
   migrations já rodaram.
3. **Ordenada.** Migrations têm `id` numérico crescente e rodam na ordem da lista. Uma vez "aplicada"
   (linha em `schema_migrations`), não roda de novo — mas como cada migration é *também* idempotente
   por construção, mesmo que `schema_migrations` seja perdida, re-rodar é seguro.
4. **Roda dentro do `Database.__init__`**, após `executescript(SCHEMA)` (que cria as tabelas legadas).
   Como `get_db` cria um `Database` por request, `apply_migrations` é chamado com muita frequência —
   por isso precisa ser **barato quando já aplicado**: a 1ª query é `SELECT id FROM schema_migrations`,
   e migrations já registradas são puladas sem tocar o schema.
5. **Seguro em produção.** O banco de produção já tem `accounts`/`transactions`/`pluggy_items` com dados
   reais. As migrations só **adicionam** estrutura. `ALTER TABLE ADD COLUMN` em SQLite é O(1) (não
   reescreve a tabela) e colunas novas nascem `NULL`/com `DEFAULT` — nenhum dado existente é perdido.
   O backfill de `reference_month` (migration 0007) é um `UPDATE` que só preenche linhas com valor nulo.

> **Por que `schema_migrations` *e* guards idempotentes?** O registro evita custo (não reprocessa a
> cada `Database()`), e os guards garantem segurança caso a tabela de controle seja apagada, o banco
> seja restaurado de um backup antigo, ou uma migration seja reordenada. Cinto e suspensório — o
> primer pede explicitamente "rodar 2× não falha" e "funciona na base de produção existente sem perder
> dados".

### 3.2 Estrutura de `src/storage/migrations.py`

```python
# src/storage/migrations.py
"""Migrations idempotentes para o schema do deon-fin.

Chamado por Database.__init__ após executescript(SCHEMA). Cada migration é uma
função (conn) -> None que SÓ adiciona estrutura (CREATE TABLE IF NOT EXISTS,
CREATE INDEX IF NOT EXISTS, ALTER TABLE ADD COLUMN guardado por PRAGMA).
Nunca destrutiva. Idempotente: rodar N vezes é seguro.
"""
from __future__ import annotations

import sqlite3
from typing import Callable

Migration = Callable[[sqlite3.Connection], None]


# -------------------------------------------------------------- helpers
def _column_exists(conn: sqlite3.Connection, table: str, column: str) -> bool:
    rows = conn.execute(f"PRAGMA table_info({table})").fetchall()
    return any(r[1] == column for r in rows)  # r[1] = nome da coluna


def _add_column(conn: sqlite3.Connection, table: str, column: str, ddl: str) -> None:
    """ALTER TABLE ADD COLUMN guardado: só roda se a coluna ainda não existe."""
    if not _column_exists(conn, table, column):
        conn.execute(f"ALTER TABLE {table} ADD COLUMN {column} {ddl}")


def _ensure_migrations_table(conn: sqlite3.Connection) -> None:
    conn.execute(
        """
        CREATE TABLE IF NOT EXISTS schema_migrations (
            id          INTEGER PRIMARY KEY,
            name        TEXT NOT NULL,
            applied_at  TEXT NOT NULL DEFAULT (datetime('now'))
        )
        """
    )


def _applied_ids(conn: sqlite3.Connection) -> set[int]:
    _ensure_migrations_table(conn)
    return {r[0] for r in conn.execute("SELECT id FROM schema_migrations").fetchall()}


# -------------------------------------------------------------- migrations
def m0001_tx_bucket_columns(conn: sqlite3.Connection) -> None:
    _add_column(conn, "transactions", "bucket_id", "INTEGER")
    _add_column(conn, "transactions", "bucket_source", "TEXT")  # 'auto'|'manual'|'rule'

def m0002_tx_tag_column(conn: sqlite3.Connection) -> None:
    _add_column(conn, "transactions", "tag_id", "INTEGER")

def m0003_tx_reference_month(conn: sqlite3.Connection) -> None:
    _add_column(conn, "transactions", "reference_month", "TEXT")  # 'YYYY-MM'
    conn.execute(
        "CREATE INDEX IF NOT EXISTS idx_tx_reference_month "
        "ON transactions(reference_month)"
    )

def m0004_tx_hidden_note(conn: sqlite3.Connection) -> None:
    _add_column(conn, "transactions", "hidden", "INTEGER NOT NULL DEFAULT 0")
    _add_column(conn, "transactions", "note", "TEXT")

def m0005_budget_buckets(conn: sqlite3.Connection) -> None:
    conn.execute(
        """
        CREATE TABLE IF NOT EXISTS budget_buckets (
            id            INTEGER PRIMARY KEY,
            key           TEXT UNIQUE NOT NULL,
            name          TEXT NOT NULL,
            color         TEXT,
            planned_kind  TEXT,                       -- 'percent' | 'amount'
            planned_value REAL,
            sort_order    INTEGER NOT NULL DEFAULT 0,
            is_system     INTEGER NOT NULL DEFAULT 1
        )
        """
    )

def m0006_tags(conn: sqlite3.Connection) -> None:
    conn.execute(
        """
        CREATE TABLE IF NOT EXISTS tags (
            id         INTEGER PRIMARY KEY,
            name       TEXT UNIQUE NOT NULL,
            color      TEXT,
            created_at TEXT NOT NULL DEFAULT (datetime('now'))
        )
        """
    )

def m0007_bucket_rules(conn: sqlite3.Connection) -> None:
    conn.execute(
        """
        CREATE TABLE IF NOT EXISTS bucket_rules (
            id         INTEGER PRIMARY KEY,
            match_key  TEXT UNIQUE NOT NULL,
            bucket_id  INTEGER REFERENCES budget_buckets(id),
            created_at TEXT NOT NULL DEFAULT (datetime('now'))
        )
        """
    )

def m0008_profile(conn: sqlite3.Connection) -> None:
    conn.execute(
        """
        CREATE TABLE IF NOT EXISTS profile (
            id                        INTEGER PRIMARY KEY CHECK (id = 1),
            name                      TEXT,
            email                     TEXT,
            monthly_income            REAL,
            financial_month_start_day INTEGER NOT NULL DEFAULT 1,
            goals_text                TEXT,
            updated_at                TEXT NOT NULL DEFAULT (datetime('now'))
        )
        """
    )

def m0009_account_balances(conn: sqlite3.Connection) -> None:
    conn.execute(
        """
        CREATE TABLE IF NOT EXISTS account_balances (
            account_id   TEXT PRIMARY KEY REFERENCES accounts(id),
            balance      REAL,
            credit_limit REAL,
            used         REAL,
            available    REAL,
            last_sync_at TEXT,
            sync_status  TEXT,
            updated_at   TEXT NOT NULL DEFAULT (datetime('now'))
        )
        """
    )

def m0010_backfill_reference_month(conn: sqlite3.Connection) -> None:
    """Backfill: preenche reference_month das linhas existentes com o mês de
    posted_at (start_day=1). O recálculo por competência real (quando o usuário
    define financial_month_start_day != 1) é feito pelo endpoint/CLI de recompute
    (ver §6/§7). Só toca linhas ainda nulas — idempotente."""
    conn.execute(
        """
        UPDATE transactions
           SET reference_month = substr(posted_at, 1, 7)
         WHERE reference_month IS NULL
           AND posted_at IS NOT NULL
        """
    )


# Ordem é o contrato. Nunca reordenar IDs já liberados; só anexar no fim.
MIGRATIONS: list[tuple[int, str, Migration]] = [
    (1,  "tx_bucket_columns",        m0001_tx_bucket_columns),
    (2,  "tx_tag_column",            m0002_tx_tag_column),
    (3,  "tx_reference_month",       m0003_tx_reference_month),
    (4,  "tx_hidden_note",           m0004_tx_hidden_note),
    (5,  "budget_buckets",           m0005_budget_buckets),
    (6,  "tags",                     m0006_tags),
    (7,  "bucket_rules",             m0007_bucket_rules),
    (8,  "profile",                  m0008_profile),
    (9,  "account_balances",         m0009_account_balances),
    (10, "backfill_reference_month", m0010_backfill_reference_month),
]


def apply_migrations(conn: sqlite3.Connection) -> int:
    """Aplica migrations pendentes na ordem. Retorna quantas rodaram nesta chamada.
    Idempotente: migrations já registradas em schema_migrations são puladas, e
    cada migration é, por construção, segura para re-rodar."""
    applied = _applied_ids(conn)
    ran = 0
    for mid, name, fn in MIGRATIONS:
        if mid in applied:
            continue
        fn(conn)
        conn.execute(
            "INSERT INTO schema_migrations (id, name) VALUES (?, ?)", (mid, name)
        )
        ran += 1
    conn.commit()
    return ran
```

### 3.3 Wiring em `Database.__init__`

```python
# src/storage/db.py (trecho)
from .migrations import apply_migrations   # novo import

class Database:
    def __init__(self, path: Path | str) -> None:
        self.path = Path(path)
        self.path.parent.mkdir(parents=True, exist_ok=True)
        self._conn = sqlite3.connect(str(self.path), check_same_thread=False)
        self._conn.row_factory = sqlite3.Row
        self._conn.executescript(SCHEMA)     # tabelas legadas (idempotente)
        apply_migrations(self._conn)         # <-- NOVO: schema novo, idempotente
        self._conn.commit()
```

> O `SCHEMA` legado (que cria `accounts`/`transactions`/`pluggy_items`) **não muda**. As 6 colunas novas
> de `transactions` poderiam ir direto no `SCHEMA`, mas o primer pede explicitamente que **colunas novas
> entrem via migration guardada por `PRAGMA`** — isso garante que bancos de produção criados antes da
> F0.1 (que já têm a tabela `transactions` sem essas colunas) recebam as colunas via `ALTER TABLE`, e não
> dependam de recriação. Manter o caminho único (migrations) também evita drift entre `SCHEMA` e migrations.

### 3.4 DDL consolidada (referência rápida)

**Colunas novas em `transactions`** (primer §4.2):

| coluna | DDL | semântica |
|---|---|---|
| `bucket_id` | `INTEGER` (FK lógica → `budget_buckets(id)`) | a "Meta" (1 dos 6 potes) |
| `bucket_source` | `TEXT` | `'auto' \| 'manual' \| 'rule'` |
| `tag_id` | `INTEGER` (FK lógica → `tags(id)`) | tag única exibida |
| `reference_month` | `TEXT` `'YYYY-MM'` | mês de competência (§7) |
| `hidden` | `INTEGER NOT NULL DEFAULT 0` | "Ocultar dos Relatórios" |
| `note` | `TEXT` | observação do usuário |

> **FKs:** SQLite só impõe FK se `PRAGMA foreign_keys=ON` (desligado por default e não ativado pelo
> `Database` atual). Mantemos as referências como **documentação/intenção** (e nas tabelas novas via
> `REFERENCES`), mas a integridade é garantida na camada de repositório (valida `bucket_id`/`tag_id`
> existentes antes do `UPDATE`). Não ativamos `foreign_keys=ON` nesta spec para não arriscar quebrar
> inserts legados que dependem do comportamento atual; se um dia ativarmos, será em migration própria
> com auditoria.

**Tabelas novas** — DDL exata nos blocos `m0005`–`m0009` acima (§3.2). Resumo de chaves/índices:

- `budget_buckets(key UNIQUE)` — seed de 6 linhas em F1.1.
- `tags(name UNIQUE)` — seed de 7 linhas em F1.2.
- `bucket_rules(match_key UNIQUE)` — regras aprendidas (F1.1).
- `profile(id CHECK(id=1))` — linha única (F2.7).
- `account_balances(account_id PK → accounts)` — snapshot de saldo/limite (F2.5).
- Índice novo: `idx_tx_reference_month ON transactions(reference_month)` (filtros por competência).

## 4. Camada de dados / repositórios

### 4.1 Organização

Pacote `src/web/repositories/` (escolha justificada em §10.3). Cada módulo expõe **funções puras** que
recebem uma `Database` (ou `sqlite3.Connection`) e isolam o SQL. **Os routers nunca escrevem SQL**;
chamam o repositório. Repositórios podem reaproveitar métodos já existentes de `Database`
(`list_transactions`, `list_accounts`, etc.) e adicionar os novos (filtros por `reference_month`,
joins com `budget_buckets`/`tags`, escrita parcial de transação).

```
  transactions_repo.py   # leitura/escrita de transactions (+ joins bucket/tag)
  accounts_repo.py       # accounts + account_balances
  buckets_repo.py        # budget_buckets + bucket_rules
  tags_repo.py           # tags
  profile_repo.py        # profile (linha única id=1)
```

> **Convivência com `Database`:** `src/storage/db.py` continua dono do *schema* e dos métodos de baixo
> nível já usados pelo domínio (`insert_transactions`, `upsert_account`, etc.) — não os movemos, para
> não quebrar `context.py`/`cards.py`/importers. Os repositórios são uma camada **acima**, específica da
> API nova. Métodos de leitura/escrita que hoje não existem (ex.: `update_transaction_fields`) podem ser
> adicionados em `Database` e **chamados** pelo repositório, mantendo o SQL dentro de `db.py`; o
> repositório vira a fachada que os routers consomem. O critério é: **nenhum `cur.execute(...)` dentro de
> `src/web/routers/`**.

## 5. API

### 5.1 Estrutura de routers

Refatorar `create_app()` para registrar `APIRouter`s por domínio. Cada arquivo declara
`router = APIRouter(prefix="/api", tags=[...])` e é incluído na factory com `app.include_router(...)`.

```
src/web/routers/
  __init__.py
  transactions.py   # /api/transactions ...           (corpo: F2.2)
  accounts.py       # /api/accounts, /api/contas ...   (corpo: F2.5)
  buckets.py        # /api/buckets ...                 (corpo: F1.1/F2.6)
  tags.py           # /api/tags ...                    (corpo: F1.2)
  budget.py         # /api/orcamento ...               (corpo: F2.3)
  invoices.py       # /api/faturas (envolve cards.py)  (corpo: F2.4)
  dashboard.py      # /api/painel (novo) ...           (corpo: F2.1)
  profile.py        # /api/profile ...                 (corpo: F2.7)
  sync.py           # /api/connect-token, /api/items, /api/sync-* (MOVE os atuais)
  legacy.py         # /, /api/summary, /api/dashboard, /api/cartao, /api/maintenance,
                    #   /api/simular, /api/amortizacao, /api/analyze  (MOVE sem alterar shape)
```

**Decisão de migração de endpoints existentes:** os endpoints atuais de `app.py` (sync Pluggy,
`/api/summary`, `/api/dashboard`, `/api/cartao`, manutenção, simulador, analyze, `/`) são **movidos**
para `routers/sync.py` e `routers/legacy.py` **sem mudar rota nem shape** (primer §3.3 e §6: não quebrar
o front legado). A factory passa a apenas montar middleware + routers. Isso é refator de organização,
não de comportamento — coberto por testes de regressão (§11).

> **Stubs:** routers de telas ainda não escritas (`transactions`, `buckets`, `tags`, `budget`,
> `invoices`, `dashboard`, `profile`) nascem em F0.1 **vazios ou com no máximo 1 endpoint trivial**
> (ex.: `GET /api/buckets` → `buckets_repo.list_buckets(db)`, que retornará `[]` até o seed de F1.1).
> O importante é o **esqueleto registrado sob `/api`** e o handler de erro ativo. Nenhum stub deve

### 5.2 Handler global de erros

Contrato (primer §6): toda resposta de erro é `{ "error": { "code": string, "message": string } }` com
status HTTP adequado. Hoje o app usa `HTTPException(detail=...)`, que produz `{"detail": "..."}`. F0.1
adiciona handlers que **normalizam** para o shape novo, **sem** alterar o `/api/health` nem o middleware
de Basic Auth (que já responde `{"detail": ...}` no 401 — ver §10.2 sobre alinhar).

```python
# src/web/errors.py
from fastapi import FastAPI, Request
from fastapi.exceptions import RequestValidationError
from fastapi.responses import JSONResponse
from starlette.exceptions import HTTPException as StarletteHTTPException

def _envelope(code: str, message: str) -> dict:
    return {"error": {"code": code, "message": message}}

_STATUS_CODE = {
    400: "bad_request", 401: "unauthorized", 403: "forbidden",
    404: "not_found", 409: "conflict", 422: "validation_error",
    502: "upstream_error", 500: "internal_error",
}

def install_error_handlers(app: FastAPI) -> None:
    @app.exception_handler(StarletteHTTPException)
    async def _http(request: Request, exc: StarletteHTTPException):
        code = _STATUS_CODE.get(exc.status_code, "error")
        message = exc.detail if isinstance(exc.detail, str) else "erro"
        return JSONResponse(
            _envelope(code, message), status_code=exc.status_code,
            headers=getattr(exc, "headers", None),
        )

    @app.exception_handler(RequestValidationError)
    async def _validation(request: Request, exc: RequestValidationError):
        return JSONResponse(
            _envelope("validation_error", "payload inválido"),
            status_code=422,
        )

    @app.exception_handler(Exception)
    async def _unhandled(request: Request, exc: Exception):
        # log no servidor; mensagem genérica pro cliente (não vazar stack/detalhe)
        return JSONResponse(_envelope("internal_error", "erro interno"), status_code=500)
```

Exemplo de resposta de erro (404):

```json
{ "error": { "code": "not_found", "message": "transação não encontrada" } }
```

Status usados: `400` bad_request, `401` unauthorized, `403` forbidden, `404` not_found, `409` conflict,
`422` validation_error, `502` upstream_error (ex.: falha Pluggy), `500` internal_error.

> **Compatibilidade do front legado:** o `app.js` legado consome `/api/summary|dashboard|cartao` (que
> retornam **200** com dados) e trata erros de forma tolerante. Mudar o shape do **erro** (de `detail`
> para `error.code/message`) não afeta o caminho feliz desses endpoints. Ainda assim, os testes de
> regressão (§11) confirmam que os 200 não mudaram.

- `src/config.py:142` → `app_password = (os.environ.get("APP_PASSWORD") or "").strip() or None`. Ou seja,
  **`APP_PASSWORD` vazio ⇒ `app_password is None`**.

Portanto **o comportamento que o primer pede ("Basic Auth desligado quando `APP_PASSWORD` vazio") já
existe** — F0.1 só o documenta e o preserva. Decisões:

- **Dev:** `APP_PASSWORD` vazio (ou ausente) no ambiente do backend ⇒ sem auth. O front Next em
  `localhost:3000` chama `http://127.0.0.1:8000/api/...` livremente (CORS liberado em §5.3). Esta é a
  configuração recomendada para desenvolvimento local — simples e sem fricção.
- **Prod:** `APP_PASSWORD` definido ⇒ Basic Auth protege tudo exceto `/api/health`. O front Next deve
  enviar o header `Authorization: Basic base64(user:pass)`. Estratégias (escolha do spec de deploy F0.2/F2):
  1. **Mesma origem + reverse proxy:** Next servido atrás do mesmo domínio da API; o proxy (Traefik/nginx)
     injeta/repassa o Basic Auth, ou o navegador o mantém após o primeiro 401 (`WWW-Authenticate`).
  2. **Credenciais no servidor Next:** chamadas à API feitas por Route Handlers/Server Components do Next,
     que adicionam o header a partir de variáveis de ambiente do servidor (a senha não vai ao browser).
  3. **`fetch` com `credentials`:** se chamado do browser, usar `credentials: 'include'` — por isso
     `allow_credentials=True` no CORS.

> **Importante (escopo):** F0.1 **não** troca Basic Auth por outro esquema (cookies/JWT). Mantém o
> mecanismo atual e deixa o `Settings.cors_origins`/`allow_credentials` prontos. O modelo de autenticação
> do front em produção é decidido em F0.2 (scaffold) / spec de deploy. App é uso pessoal/familiar, 1
> perfil (primer §5.5) — não há multiusuário.

## 6. Utils — `reference_month` e helpers de sinal

### 6.1 `reference_month(posted_at, start_day) -> 'YYYY-MM'`

Implementa a competência do primer §5.4: se o mês financeiro começa no dia `start_day`, um lançamento
em `posted_at` pertence ao mês `'YYYY-MM'` cujo ciclo o contém. Regra: **se `dia(posted_at) >= start_day`,
a competência é o próprio mês; senão, é o mês anterior** (porque o lançamento caiu na "cauda" do ciclo
que começou no mês anterior). Com `start_day = 1`, é sempre o mês civil (`posted_at[:7]`).

```python
# src/storage/reference_month.py  (importável também por web/utils e importers)
from __future__ import annotations
from datetime import date

def reference_month(posted_at: date | str, start_day: int = 1) -> str:
    """Mês de competência 'YYYY-MM' para uma data, dado o dia de início do mês
    financeiro (profile.financial_month_start_day).

    start_day=1  -> mês civil.
    start_day=15 -> 15/jun..14/jul pertencem a '<ano>-06'.
    """
    d = date.fromisoformat(posted_at) if isinstance(posted_at, str) else posted_at
    sd = max(1, min(28, int(start_day)))   # clamp p/ evitar buracos em meses curtos
    y, m = d.year, d.month
    if d.day < sd:
        # caiu antes do início do ciclo -> pertence ao mês anterior
        m -= 1
        if m == 0:
            m = 12
            y -= 1
    return f"{y:04d}-{m:02d}"
```

**Exemplos de borda (viram casos de teste — ver §11):**

| posted_at | start_day | reference_month | porquê |
|---|---|---|---|
| `2026-06-20` | 1 | `2026-06` | mês civil |
| `2026-06-14` | 15 | `2026-05` | dia 14 < 15 → ciclo começou em 15/mai |
| `2026-06-15` | 15 | `2026-06` | dia 15 = início do ciclo de junho |
| `2026-06-30` | 15 | `2026-06` | cauda de junho |
| `2026-07-14` | 15 | `2026-06` | ainda no ciclo iniciado em 15/jun |
| `2026-01-10` | 15 | `2025-12` | **vira de ano** para trás |
| `2026-12-20` | 15 | `2026-12` | dezembro, dia ≥ 15 |
| `2027-01-05` | 15 | `2026-12` | **vira de ano**: 5/jan < 15 → dez/2026 |
| `2026-02-27` | 28 | `2026-01` | clamp: start_day 28; 27 < 28 → jan |

> **Clamp em 28:** `start_day` é limitado a `[1, 28]` porque dias 29–31 não existem em todos os meses
> (fevereiro), o que criaria competências ambíguas/buracos. A UI de Perfil (F2.7) deve oferecer 1–28.
> Documentar isso é parte da reconciliação; se o primer permitir 1–31, mantemos clamp e registramos a
> divergência (§10.5).

### 6.2 Onde materializar `transactions.reference_month`

1. **Na importação (caminho novo):** sempre que transações entram (Pluggy, OFX, CSV), preencher
   `reference_month` com `reference_month(posted_at, start_day_atual)`. O `start_day` vem de
   `profile.financial_month_start_day` (default 1 enquanto o Perfil não foi configurado).

   - **Hook concreto:** hoje o pós-processamento de sync já chama `Categorizer().apply_to_database(db)`
     em **`src/web/app.py`** (`_background_sync`, linha ~106; e `_sync_all_items`, linha ~147) e no CLI
     (`categorize`). F0.1 adiciona, no mesmo ponto, uma chamada
     `transactions_repo.recompute_reference_months(db, start_day)` (ou um `fill_missing_reference_months`
     que só preenche nulos) logo após a categorização. Isso evita tocar o `insert_transactions` de
     baixo nível (que é chamado por todos os importers e por testes), centralizando a materialização no
     pós-sync — barato e idempotente.
   - Alternativa considerada: setar `reference_month` dentro de `insert_transactions`. **Rejeitada** por
     acoplar o util de competência ao storage de baixo nível e exigir passar `start_day` por toda a
     cadeia de importers. O hook pós-sync é menos invasivo.

2. **No backfill (linhas existentes):** migration `m0010` (§3.2) preenche `reference_month = posted_at[:7]`

### 6.3 Recálculo quando `financial_month_start_day` muda

Quando o usuário altera o dia de início (Perfil, F2.7), **todas** as competências precisam ser
recalculadas. F0.1 fornece a rotina; o endpoint que a dispara é detalhado em F2.7, mas o contrato fica
aqui:

- **Repo:** `transactions_repo.recompute_reference_months(db, start_day) -> int` — itera as transações
  e faz `UPDATE transactions SET reference_month = ? WHERE id = ?` com o valor novo; retorna a contagem.
- **CLI:** novo comando `python -m src.cli recompute-reference-month` (lê `start_day` do `profile`, ou
  default 1) — útil para operação/manutenção em produção.
- **Endpoint (declarado em F2.7, citado aqui):** ao `PATCH` o perfil com `financial_month_start_day`
  diferente, o handler chama `recompute_reference_months`. F0.1 só garante a função e o CLI.

```python
# src/agent/context.py  (adicionar; assinaturas idênticas ao plano de financial-signs)
def spending_value(amount: float, account_type: str | None, category: str | None) -> float:
    """Impacto positivo de gasto p/ despesas; negativo p/ estornos; 0 p/ não-consumo."""
    cat = category or "(sem categoria)"
    if cat in NON_SPENDING_CATEGORIES:
        return 0.0
    value = float(amount)
    if (account_type or "").upper() in CREDIT_TYPES:
        return value                      # cartão: compra +, estorno -
    return -value if value < 0 else 0.0   # banco: débito vira gasto +

def income_value(amount: float, account_type: str | None, category: str | None) -> float:
    """Renda só de contas bancárias, excluindo transferências internas."""
    cat = category or "(sem categoria)"
    if cat in NON_SPENDING_CATEGORIES:
        return 0.0
    if (account_type or "").upper() in CREDIT_TYPES:
        return 0.0
    value = float(amount)
    return value if value > 0 else 0.0
```

## 11. Divergências entre código atual e o primer (reconciliações propostas)

1. **`spending_value`/`income_value` não existem no código.** O primer §5.3 os trata como existentes
   ("ex.: `spending_value`, `income_value`"), mas hoje só há a lógica inline em `context.py` e a
   *proposta* no plano financial-signs. **Reconciliação:** F0.1 (Bloco C) materializa esses helpers em
   `src/agent/context.py` com as assinaturas exatas do plano e refatora `build_financial_context` de
   forma comportamento-preservante. A partir daí, a API nova só usa os helpers. (Se o Sprint de
   financial-signs já tiver sido implementado quando F0.1 começar, o Bloco C vira um no-op de verificação.)

2. **Shape de erro atual ≠ contrato do primer.** O middleware Basic Auth (`app.py:204`) e os
   `HTTPException` retornam `{"detail": ...}`; o primer §6 manda `{ "error": { code, message } }`.
   **Reconciliação:** `install_error_handlers` normaliza `HTTPException`/validação/erros não tratados
   para o envelope novo. O 401 do middleware pode ser alinhado (opcional) para também emitir o envelope —
   recomendado para consistência, mas mantendo `WWW-Authenticate` e o `/api/health` liberado. Decisão:
   alinhar o 401 ao envelope (baixo risco; o front legado não depende do corpo do 401).

3. **Repositórios: `src/web/repositories/` vs `src/storage/queries.py`.** O primer §3.1 cita ambos como
   opção. **Decisão:** `src/web/repositories/` (pacote por domínio) — escala melhor com a API por
   domínio, mantém `src/storage/db.py` focado no schema/baixo nível, e deixa claro que a fachada é da
   camada web. SQL de baixo nível continua em `db.py` (os repositórios chamam métodos de `Database`).

4. **Materialização de `reference_month`: insert vs pós-sync.** O primer §5.4 diz "materializado na
   importação". **Decisão/refinamento:** materializar no **pós-sync** (junto da categorização, onde já há
   um ponto único — `app.py` `_background_sync`/`_sync_all_items` e CLI `categorize`), em vez de no
   `insert_transactions` de baixo nível, para não acoplar competência ao storage nem propagar `start_day`
   por todos os importers. O efeito prático é o mesmo (toda transação importada ganha `reference_month`),
   com menos acoplamento. Registrado como refinamento, não contradição.

5. **`financial_month_start_day` 1–31 vs clamp 1–28.** O primer não fixa o range. **Proposta:** clamp em
   `[1,28]` no util (evita competência ambígua em fevereiro). A UI de Perfil (F2.7) deve oferecer 1–28.
   Se o produto exigir 29–31, tratar como regra explícita ("último dia do mês") em spec futura.
