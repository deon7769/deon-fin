## 1. Contexto & objetivo

Hoje o app **conecta e sincroniza** contas via Pluggy, mas a UI é a aba "Conexões & Sincronização" do front
legado (`src/web/templates/index.html`), que lista apenas os *items* (`GET /api/items`) e dispara sync. **Não
há saldo persistido**: a tabela `accounts` (`src/storage/db.py`) não tem coluna de saldo, e `context.contas`
é anonimizado para `{nome, tipo}` (`src/agent/context.py`). Por isso o KPI "Saldo em conta" do Painel (F2.1)
e os limites de cartão das Faturas (F2.4) ficam sem fonte real.

A boa notícia: **o dado de saldo já chega do Pluggy** e até já é guardado de forma crua. Em
`src/importers/pluggy_sync.py`, `_sync_account(...)` recebe cada conta de `client.list_accounts(item_id)`
(`src/pluggy/client.py:128`) e faz:

```python
db.upsert_account(Account(
    id=f"pluggy:{acc['id']}", source="pluggy",
    institution=bank_data.get("transferNumber") or acc.get("name"),
    name=acc.get("name"), type=acc.get("type"), currency=acc.get("currencyCode", "BRL"),
    metadata={k: v for k, v in acc.items() if k not in {"id", "name", "type"}},  # ← inclui balance, bankData, creditData, itemId
))
```

Ou seja, `acc["balance"]`, `acc["bankData"]`, `acc["creditData"]` e `acc["itemId"]` **já estão** em
`accounts.metadata_json`. Falta **normalizar** isso em `account_balances` (criada vazia em F0.1 §4.3) e expor
uma tela rica.

**Objetivo:** (a) estender o sync para popular `account_balances`; (b) `GET /api/accounts` agregando
contas/cartões/saldos/totais; (c) embrulhar os fluxos de item existentes (sync, credenciais, remover, nova
conexão) em endpoints orientados a "conta"; (d) tela `/contas` (Cards/Lista, KPIs, tabelas, ações, estado de sync).

## 3. Backend

### 3.1 Forma da conta no Pluggy (referência p/ extração)
`client.list_accounts(item_id)` devolve, por conta, algo como:
```jsonc
{
  "id": "acc_123", "itemId": "item_xyz", "type": "BANK",        // ou "CREDIT"
  "name": "Conta Corrente", "marketingName": "Inter", "number": "31238064-0",
  "balance": 58.77, "currencyCode": "BRL",
  "bankData": { "transferNumber": "077/0001/31238064-0" },
  "creditData": { "brand": "MASTERCARD", "creditLimit": 4000.00, "availableCreditLimit": 25.65, "level": "BLACK" }
}
```

### 3.2 Preencher a lacuna de saldo (estender o sync)
**Mudança em `src/importers/pluggy_sync.py` → `_sync_account`:** após `db.upsert_account(...)`, normalizar o
snapshot via um helper de repositório (F0.1) e preencher `pluggy_item_id`:
```python
from ..agent.cards import CREDIT_TYPES          # {"CREDIT","CREDIT_CARD"} — única fonte da verdade
is_credit = (acc.get("type") or "").upper() in CREDIT_TYPES
cd = acc.get("creditData") or {}
limit, available = cd.get("creditLimit"), cd.get("availableCreditLimit")
used = (limit - available) if (limit is not None and available is not None) else cd.get("balance")

accounts_repo.upsert_balance(
    db, account_id=acc_id,
    balance=None if is_credit else acc.get("balance"),
    credit_limit=limit, used=used, available=available,
    brand=cd.get("brand"), last4=_last4(acc.get("number")),
    last_sync_at=utcnow_iso(),
    sync_status=(db.get_pluggy_item(item_id) or {}).get("status") or "UPDATED",
)
# também: db.set_account_item(acc_id, acc.get("itemId"))  → preenche accounts.pluggy_item_id
```
- O sync já roda em `sync_pluggy_item` (chamado por `_background_sync`/`_sync_all_items` em `src/web/app.py` e
  pela CLI) — **nenhuma orquestração nova**; só o upsert por conta.
- **Backfill** (sem novo sync): rotina/migration única que lê `accounts.metadata_json` das contas
  `source='pluggy'` e popula `account_balances` a partir de `metadata.balance`/`metadata.creditData`.
  Idempotente. Cobre a base de produção sem esperar o próximo ciclo.
- **`sync_status`** reflete `pluggy_items.status` (`UPDATED|OUTDATED|LOGIN_ERROR|…`). Em `LOGIN_ERROR`,
  manter último saldo e sinalizar na UI.
- **Contas manuais (OFX/CSV):** sem saldo do provedor → derivar `balance = Σ amount` da conta e marcar
  `sync_status='DERIVED'` (gancho p/ edição manual futura). Cartão manual sem `creditData` → limite/uso `NULL`.

### 3.3 `accounts_repo` (sem SQL nos routers)
```python
def list_banks(db) -> list[dict]: ...      # accounts.type ∉ CREDIT_TYPES ⨝ account_balances ⨝ pluggy_items
def list_cards(db) -> list[dict]: ...      # accounts.type ∈ CREDIT_TYPES  ⨝ account_balances
def totals(db, *, month: str) -> dict: ... # {accounts_balance, card_debt, period_result}
def upsert_balance(db, *, account_id, balance, credit_limit, used, available, brand, last4,
                   last_sync_at, sync_status) -> None: ...
def resolve_item_id(db, account_id: str) -> str | None: ...   # accounts.pluggy_item_id (fallback metadata.itemId)
def set_sort(db, order: list[str]) -> None: ...
def disconnect(db, item_id: str, *, keep_transactions: bool = True) -> dict: ...  # ver §3.6
```
`upsert_balance` faz `INSERT ... ON CONFLICT(account_id) DO UPDATE` em `account_balances`.

### 3.4 Mapeamento conta ↔ item
Um *item* (conexão) tem **N contas**. Hoje a relação vive só em `accounts.metadata_json.itemId`. Para as ações
por conta, adicionamos **`accounts.pluggy_item_id`** (migration aditiva), preenchida no `_sync_account`, com
fallback de leitura em `metadata_json.itemId` para registros antigos. As ações de sync/credenciais/remover
operam no **item** (a API Pluggy é por item); a UI deixa explícito quando o item tem várias contas.

### 3.5 `GET /api/accounts`
```jsonc
// GET /api/accounts?month=YYYY-MM   (month só p/ period_result; default = competência atual)
{
  "banks": [
    { "id": "pluggy:acc_1", "institution": "BANCO INTER", "name": "Conta corrente",
      "type": "Conta corrente", "agency": "077/0001", "number": "31238064-0",
      "balance": 58.77, "currency": "BRL",
      "pluggy_item_id": "item_inter", "connector_name": "Banco Inter",
      "last_sync_at": "2026-06-19T13:47:00", "sync_status": "UPDATED", "manual": false }
  ],
  "cards": [
    { "id": "pluggy:acc_2", "name": "Cartão BTG BLACK", "last4": "2970", "brand": "MASTERCARD",
      "credit_limit": 4000.00, "used": 0.00, "available": 25.65, "usage_pct": 0.0,
      "pluggy_item_id": "item_btg", "last_sync_at": "2026-06-19T20:43:00", "sync_status": "UPDATED" }
  ],
  "totals": { "accounts_balance": 67.67, "card_debt": 0.00, "period_result": 67.67 },
  "sync": { "running": false, "last_finished": "2026-06-19T20:43:10", "last_result": "2 conta(s) sincronizada(s)" }
}
```
- `agency`/`number` derivados de `bankData.transferNumber` (split por `/`). `usage_pct = used/credit_limit*100`
  (null sem limite). `last4` dos últimos 4 dígitos de `number`/`marketingName`.
- `totals.accounts_balance` = Σ saldos de banco; `card_debt` = Σ `used` dos cartões; `period_result` =
  receitas−despesas do período — **reusar o mesmo helper** do `GET /api/painel/summary` (F2.1), não recomputar.
- `sync` espelha `GET /api/sync-status` (`_sync_state` em `app.py`) p/ a UI mostrar "Sincronizando…".
- Sem contas → listas vazias e totais 0 (200, não erro). Erros no envelope `{error:{code,message}}`.

### 3.6 Ações (embrulham os endpoints de item)

| Endpoint novo | Reusa | Observação |
|---|---|---|
| `POST /api/accounts/{id}/sync` | `_background_sync(item_id, days)` | resolve `id→item`; 404 conta desconhecida; 409 se sync em andamento |
| `POST /api/accounts/{id}/credentials` | `POST /api/connect-token {item_id}` | retorna `{accessToken}` p/ reabrir o widget em modo *update* |
| `DELETE /api/accounts/{id}` | `DELETE /api/items/{item_id}` | **mantém as transações** (decisão fixada) — ver abaixo |
| `PATCH /api/accounts/sort` | — | body `{ order: [account_id, …] }`; persiste `sort_order` |
| "Nova conta" | `POST /api/connect-token` + `POST /api/items` | fluxo do widget; nada novo no backend |

**Remover conexão — comportamento definitivo (`keep_transactions=True`):**
`DELETE /api/accounts/{id}` resolve o `item_id` e:
1. Chama `client.delete_item(item_id)` no Pluggy (como o `DELETE /api/items/{id}` atual; tolera 404).
2. `DELETE FROM pluggy_items WHERE id=item_id`.
3. **NÃO** apaga `transactions` nem `accounts` das contas do item — o histórico financeiro é preservado.
   Marca as `accounts` afetadas como **desconectadas** (`account_balances.sync_status='DISCONNECTED'`) para a UI
   poder rotular "conta desconectada (histórico mantido)". Saldo permanece como último snapshot.
Resposta: `{ deleted: true, item_id, kept_transactions: true, accounts_disconnected: [ids] }`.
> Sem opção destrutiva nesta versão. (Se um dia for desejada uma "limpeza", entra como ação avançada e
> separada — fora do escopo.)

### 4.3 Comportamento
- Hook `useAccounts(month)` (key `['accounts', month]`), `month` de `usePeriod` só p/ `period_result`.
- Mutations: `useSyncAccount`, `useUpdateCredentials` (abre o widget Pluggy em modo update com o `accessToken`),
  `useDeleteAccount` (confirm: "Isso remove a **conexão** e suas N contas do provedor. **As transações já
  importadas serão mantidas.**"), `useReorderAccounts` (dnd → `PATCH /sort`, optimistic).
- **Estado de sync:** ao disparar, *poll* de `GET /api/sync-status` (ou `sync` em `/api/accounts`) até
  `running=false`; spinner na linha; ao concluir, invalida `['accounts']`.
- **"+ Nova conta":** carrega o Pluggy Connect (script CDN, como no legado), chama `POST /api/connect-token`,
  no callback chama `POST /api/items` e invalida `['accounts']`.
- Estados: loading (skeleton), **vazio** (nenhuma conta → `EmptyState` "Conectar banco"), erro/retry.
  `LOGIN_ERROR`/`DISCONNECTED` destacam a linha com CTA "Atualizar credenciais"/"Reconectar".

## 5. Reaproveitamento e consistência
- **`CREDIT_TYPES`** importado de `src/agent/cards.py` (uma fonte da verdade p/ "é cartão?").
- **`period_result`** e o KPI "Saldo em conta" (F2.1) usam **a mesma** função (`accounts_repo.totals`/helper de
  saldo do Painel).
- **F2.4 (Faturas)** lê `account_balances`/metadata p/ limite/bandeira/final — nomes idênticos
  (`credit_limit`, `available`, `last4`, `brand`).

## 6. Regras / edge cases
- **Item com várias contas:** sync/remover agem no item; a UI explicita "afeta N contas".
- **Remover conexão:** transações **mantidas** (decisão fixada); contas marcadas `DISCONNECTED`.
- **`usage_pct` sem limite:** `null` → barra "—".
- **Sync em andamento:** `POST /sync` retorna `{scheduled:false, detail:"já em andamento"}` (espelha `/api/sync-all`).
- **`LOGIN_ERROR`/`OUTDATED`:** manter último saldo; chip vermelho/amarelo; CTA credenciais.
- **Conta manual:** sem item → sem sync/credentials; saldo `DERIVED`.
- **Divergência do print** (Limite 4.000 / Utilizado 0 / Disponível 25,65): dado do provedor; a UI só exibe.
