## 1. Contexto & objetivo

No print, "Perfil" tem o card **"Informações Pessoais"** ("Atualize suas informações de perfil") com: avatar
de iniciais (ex.: "DA"), **Nome**, **E-mail**, **Renda Mensal** (R$), **Início do Mês Financeiro** (ex.: 15) e
**Objetivos Financeiros** (textarea), além de **Salvar**.

Hoje esses dados vivem espalhados: `MONTHLY_INCOME`/`FINANCIAL_GOALS` no `.env` (lidos em `src/config.py` →
`settings.monthly_income`, `settings.financial_goals`) e `data/family_profile.json` (receitas/metas),
manipulado por `src/agent/maintenance.py` (`load_family_profile`, `save_family_profile`, `income_from_profile`)
e pelo `GET/POST /api/maintenance`. **Não há tela amigável** para editar nome/e-mail/renda/início do mês.

F0.1 cria a tabela **`profile`** (linha única `id=1`): `name`, `email`, `monthly_income`,
`financial_month_start_day` (default 1), `goals_text`, `updated_at`. Esta spec a torna a **fonte canônica**
desses campos e constrói a tela. Efeito colateral importante: mudar **"Início do Mês Financeiro"** recalcula a
competência (`reference_month`) das transações.

## 2. Escopo
### 2.1 Dentro do escopo
- `GET /api/profile` e `PUT /api/profile`.
- Auto-criação/seed da linha `profile` a partir das fontes legadas no primeiro acesso (migração suave).
- Recálculo de `transactions.reference_month` quando `financial_month_start_day` muda (reusa a rotina de F0.1).
- Tela `/perfil` com formulário validado, avatar de iniciais e feedback.
### 2.2 Fora do escopo
- Multiusuário/login (app é pessoal — primer §5.5). Auth segue o Basic Auth opcional.
- A aba "Manutenção" (`/api/maintenance`) — mantida (ver §3.4).
- Upload de foto de avatar (usar iniciais).

## 3. Backend

### 3.1 `profile_repo` (em `src/web/repositories/profile.py`)
```python
DEFAULTS = {"name": "", "email": "", "monthly_income": 0.0,
            "financial_month_start_day": 1, "goals_text": ""}

def get_or_create(db) -> dict:
    row = _select_profile(db)              # SELECT * FROM profile WHERE id=1
    if row:
        return dict(row)
    # seed a partir das fontes legadas (uma vez)
    from ..config import settings
    from ..agent import maintenance as mnt
    fam = mnt.load_family_profile()
    seed = {**DEFAULTS,
            "monthly_income": settings.monthly_income or mnt.income_from_profile(fam) or 0.0,
            "goals_text": ", ".join(settings.financial_goals or [])}
    _insert_profile(db, id=1, **seed)
    return {"id": 1, **seed}

def update(db, **fields) -> dict: ...      # UPSERT id=1, set updated_at=datetime('now')
```

### 3.2 `GET /api/profile`
```jsonc
{
  "name": "Davi De Oliveira Neto",
  "email": "davioliveira77@gmail.com",
  "monthly_income": 0.0,
  "financial_month_start_day": 15,
  "goals_text": "Quitar o financiamento; reserva de 6 meses",
  "initials": "DA",                 // derivado do nome
  "updated_at": "2026-06-20T10:00:00"
}
```
Se a linha não existe, `get_or_create` cria com defaults derivados das fontes legadas e retorna.

### 3.3 `PUT /api/profile`
```jsonc
// body
{ "name": "...", "email": "...", "monthly_income": 10490.41,
  "financial_month_start_day": 15, "goals_text": "..." }
// resposta
{ "saved": true, "reference_month_recompute": "scheduled" }  // ou "not_needed"
```
Implementação (router `src/web/routers/profile.py`):
```python
@router.put("/api/profile")
def update_profile(body: ProfileIn, bg: BackgroundTasks, db: Database = Depends(get_db)):
    current = profile_repo.get_or_create(db)
    validate(body)                                   # 422 no envelope {error:{code,message}}
    day_changed = body.financial_month_start_day != current["financial_month_start_day"]
    profile_repo.update(db, **body.model_dump())
    if day_changed:
        # reusa a rotina de F0.1 (mesma do comando CLI recompute-reference-month)
        bg.add_task(recompute_reference_month_all, settings.database_path,
                    start_day=body.financial_month_start_day)
    return {"saved": True, "reference_month_recompute": "scheduled" if day_changed else "not_needed"}
```
Validações (`ProfileIn` / `validate`): e-mail em formato válido se informado; `financial_month_start_day ∈
[1,28]` (alinha com o clamp de `reference_month`, F0.1 §6.1); `monthly_income ≥ 0`.

- **Efeito (renda):** a nova `monthly_income` passa a valer para Orçamento (F2.3)/Painel via a **precedência já
  definida em F2.3** (receitas do mês → `profile.monthly_income` → `settings` → `income_from_profile`). Aqui só
  persistimos.
- **Efeito (competência):** mudar o dia dispara o recálculo de `transactions.reference_month` em
  `BackgroundTasks` (não bloqueia a resposta). Idempotente e seguro (só atualiza `reference_month`).

### 3.4 Reconciliação com fontes legadas (sem quebrar)
- `profile` é canônica p/ os 5 campos da tela. `settings.monthly_income`/`financial_goals` (do `.env`) viram
  **fallback** quando `profile` está vazio.
- `data/family_profile.json` e `/api/maintenance` continuam para o perfil familiar rico
  (receitas/patrimônio/provisões/wishlist) usado por `context.py`/dashboard executivo. **Decisão:** salvar a
  renda no `profile` **não** sobrescreve `family_profile.json` (evita efeito colateral no dashboard
  executivo); se o usuário usa o perfil familiar, a renda efetiva pode vir dele (precedência de F2.3). Junção
  fica como follow-up documentado.

## 4. Frontend (`/perfil`)

```tsx
// web/app/perfil/page.tsx (resumo)
export default function PerfilPage() {
  const { data, isLoading } = useProfile();
  const save = useUpdateProfile();             // PUT; invalida ['profile'] e, se dia mudou, ['budget','transactions','painel']
  const form = useForm<ProfileForm>({ values: data });
  return (
    <SectionCard title="Informações Pessoais" subtitle="Atualize suas informações de perfil.">
      <div className="flex gap-6">
        <Avatar initials={initialsOf(form.watch("name"))} />   {/* 2 iniciais; fundo accent */}
        <form onSubmit={form.handleSubmit((v) => save.mutate(v))} className="grid grid-cols-2 gap-4">
          <Field label="Nome"  {...form.register("name")} />
          <Field label="E-mail" type="email" {...form.register("email")} />
          <MoneyInput label="Renda Mensal" {...form.register("monthly_income")} />
          <StartDayStepper label="Início do Mês Financeiro" min={1} max={28}
            tooltip="Define o mês de competência: lançamentos a partir desse dia contam para o mês seguinte."
            {...form.register("financial_month_start_day")} />
          <Textarea className="col-span-2" label="Objetivos Financeiros" {...form.register("goals_text")} />
          <button className="col-span-2 ml-auto rounded bg-accent px-4 py-2 text-black">Salvar</button>
        </form>
      </div>
    </SectionCard>
  );
}
```
- **Avatar**: 2 primeiras iniciais do nome (`initialsOf`), fundo `accent`. Sem nome → "?".
- **MoneyInput**: entrada monetária pt-BR (máscara/parse para número antes do PUT).
- **StartDayStepper**: select/stepper 1–28 com tooltip explicando a competência.
- Ao salvar com o **dia alterado**, mostrar toast "Recalculando o mês de referência das transações…" e, ao
  concluir, invalidar `['transactions']`/`['budget']`/`['painel']`.
- `MoneyText`/Ocultar não se aplica a inputs de edição; a Renda exibida em outras telas respeita Ocultar.
- Estados: loading (skeleton do form), erro/retry, validação inline (e-mail, dia, renda).

## 5. Regras / edge cases
- **Nome vazio** → avatar "?"; saudação do Painel cai para "Olá".
- **E-mail vazio** permitido (uso pessoal); se preenchido, validar formato.
- **Mudar `financial_month_start_day`** reprocessa competência: pode reclassificar transações de borda entre
  meses — avisar o usuário; telas dependentes refazem o fetch. Operação idempotente.
- **Renda 0** é válida → Orçamento (F2.3) mostra "Defina sua renda" apontando para esta tela.
- **Concorrência:** linha única; `updated_at` p/ detectar edição externa (opcional).
- **Pré-preenchimento:** e-mail/nome podem vir do seed legado; nunca hardcodar no código.
