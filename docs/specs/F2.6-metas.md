## 1. Contexto & objetivo

O **Orçamento** (F2.3) mostra "Metas Financeiras": 6 potes com **Gasto** vs **Previsto**. Esse **Previsto** é
configurado aqui. O menu lateral tem o item **"Metas"**, e a palavra "metas" no app cobre duas coisas — ambas
entram nesta tela (decisão do usuário):

1. **Distribuição do orçamento por pote** — define `budget_buckets.planned_kind/planned_value` (criados em
   F1.1), que o Orçamento consome.
2. **Metas de poupança / objetivos** — objetivos com alvo/prazo/progresso. O código **já tem a matemática**:
   `summarize_wishlist(items, sobra_mensal)` em `src/agent/budget.py` retorna, por item, `guardar_mes`,
   `progresso_pct`, `cabe_na_sobra`, e agregados `total_guardar_mes`/`folga`. Hoje os itens vêm de
   `family_profile["wishlist"]`/`["metas"]` (lidos via `src/agent/maintenance.py`). Esta tela dá a eles uma
   UI e persistência própria.

**Objetivo:** tela `/metas` com (A) editor da distribuição por pote (com validação de soma e preview em R$) e
(B) CRUD de metas de poupança reaproveitando `summarize_wishlist`.

### 3.2 Endpoints

- `PATCH /api/buckets/{id}` (F1.1) `{ name?, color?, planned_kind?, planned_value?, sort_order? }`.

- `GET /api/buckets/plan?month=YYYY-MM` (novo, conveniência para a tela):
```jsonc
{

  "income_source": "profile",         // receitas | profile | settings | family | none
  "buckets": [
    { "id": 1, "name": "Liberdade Financeira", "color": "#22C55E",
      "planned_kind": "percent", "planned_value": 25.0,
      "planned_amount": 2622.60,      // percent→income*pct/100; amount→o valor
      "spent_month": 2223.60 }        // realizado do mês (read-only, de F2.3)
  ],
  "sum_percent": 100.0,               // soma só dos potes 'percent'
  "sum_amount": 10490.41,             // soma dos planned_amount

```
`planned_amount` deve usar **exatamente** o mesmo helper de F2.3 (uma fonte da verdade; ex.:
`budget_calc.planned_amount(bucket, income)`), para o Previsto bater entre as telas.

## 4. Backend — Bloco B (metas de poupança)

### 4.1 Migration (aditiva, idempotente — anexa à lista de F0.1)
```sql
CREATE TABLE IF NOT EXISTS savings_goals (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  name          TEXT NOT NULL,
  target_amount REAL NOT NULL,           -- valor-alvo
  term_months   INTEGER NOT NULL DEFAULT 12,
  saved_amount  REAL NOT NULL DEFAULT 0, -- já guardado
  priority      INTEGER NOT NULL DEFAULT 99,
  created_at    TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at    TEXT NOT NULL DEFAULT (datetime('now'))
);
```
**Import inicial (uma vez):** se `savings_goals` está vazia e existe `family_profile["metas"]`/`["wishlist"]`
(via `mnt.load_family_profile()`), semear a partir desses itens (mapear `nome→name`,
`valor_alvo→target_amount`, `prazo_meses→term_months`, `guardado→saved_amount`, `prioridade→priority`).
Idempotente (só roda na 1ª vez / tabela vazia).

### 4.2 Repositório e reuso de `summarize_wishlist`
`savings_repo` (CRUD) + o resumo **não reimplementa** a conta — chama `summarize_wishlist`:
```python
from ..agent.budget import summarize_wishlist
from ..agent.context import build_financial_context  # p/ media_gasto_mensal

def list_with_summary(db, *, income: float | None) -> dict:
    items = savings_repo.list(db)                       # [{nome, valor_alvo, prazo_meses, guardado, prioridade}]
    ctx = build_financial_context(db, monthly_income=income, goals=[]).to_dict()
    sobra_mensal = (income or 0) - ctx["media_gasto_mensal"]
    return summarize_wishlist(items, sobra_mensal)      # {itens:[...guardar_mes,progresso_pct,cabe_na_sobra], total_guardar_mes, folga, sobra_mensal}
```
> `summarize_wishlist` espera as chaves `nome, valor_alvo, prazo_meses, guardado, prioridade` — o repo deve
> mapear as colunas (`name→nome`, etc.) ao montar a lista, mantendo o contrato existente da função.

### 4.3 Endpoints
- `GET /api/savings-goals?month=YYYY-MM` → `{ goals:[{ id, name, target_amount, term_months, saved_amount, priority, guardar_mes, progresso_pct, cabe_na_sobra }], total_guardar_mes, folga, sobra_mensal }`.
- `POST /api/savings-goals` `{ name, target_amount, term_months, saved_amount?, priority? }` → 201.
- `PATCH /api/savings-goals/{id}` (parcial).
- `DELETE /api/savings-goals/{id}`.
Erros no envelope `{error:{code,message}}` (F0.1); validar `target_amount > 0`, `term_months ≥ 1`,
`saved_amount ≥ 0`.

export function BucketPlanRow({ b, income }: { b: BucketPlan; income: number | null }) {
  const previsto = b.planned_kind === "percent" && income != null
    ? (income * b.planned_value) / 100 : (b.planned_kind === "amount" ? b.planned_value : null);

## 6. Regras / edge cases
- **Renda 0/indefinida:** preview em R$ e `cabe_na_sobra` ficam indefinidos ("—"); o aviso de soma (%) ainda
  funciona; CTA → `/perfil`.
- **Mistura percent/amount:** a soma de 100% considera só os `percent`; `amount` é absoluto (documentar na UI).
- **`sobra_mensal ≤ 0`:** `summarize_wishlist` marca tudo como não-cabe; a UI mostra alerta "Sem sobra mensal".
- **Prioridade duplicada:** ordenação estável por (prioridade, −valor_alvo) — já é o comportamento da função.
- **Pote `is_system`:** edita previsto/cor/ordem, não exclui.
- **Consistência com F2.3:** `planned_amount` (percent→R$) idêntico ao de F2.3 (mesmo helper, mesma renda).
