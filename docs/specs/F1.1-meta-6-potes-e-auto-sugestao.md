## 1. Contexto & objetivo

O print mostra, em cada linha de transação, um seletor **"Meta"** — o usuário escolhe um dos **6 potes**
do método (Liberdade Financeira, Custos Fixos, Conforto, Metas, Prazeres, Conhecimento). O texto do print
diz: _"Transações feitas nas contas com a meta categorizada, podendo editar manualmente, **já sugerindo
nas do mesmo tipo**"_. Ou seja, ao definir a Meta de uma transação, o sistema **propaga** a mesma Meta
para outras transações **do mesmo tipo** (mesma contraparte/descrição normalizada, mesmo sinal) que ainda
estão **"Sem meta"**.

Como isso se liga ao código atual:

- O domínio já tem **dois** classificadores baratos e determinísticos que esta spec reaproveita:
  - **`src/agent/categorizer.py`** — `Categorizer`, `DEFAULT_RULES` (regex → categorias **PT granulares**
    como `"Alimentação - Restaurante"`, `"Transporte - App"`), `classify(description)` e
    `apply_to_database(db, *, overwrite_pluggy=False)`. É o mecanismo que rodamos na ingestão e que
    estendemos para também atribuir **Meta** (pote).
  - **`src/agent/budget.py`** — `BUDGET_MAP` (categoria **EN do Pluggy** + **PT local** → bloco
    `essencial/desejos/financeiro` do 50/30/20) e `classify(category)`. É o **espírito** que copiamos
    para o novo **mapa default `category → bucket`** (`src/agent/buckets.py`).
- A normalização de comerciante já existe e **deve ser reusada como chave de similaridade**:
  **`_merchant_key(raw)` em `src/agent/context.py:72`** — faz `anonymize(raw).lower()`, tokeniza via
  `_TOKEN`, descarta `_NOISE_TOKENS` e dígitos, e junta os **4 primeiros tokens**. É exatamente a chave
  que `bucket_rules.match_key` vai usar (com o sinal anexado — ver §4.4).
- O schema já foi **criado vazio em F0.1**: a coluna `transactions.bucket_id` / `transactions.bucket_source`
  (migration `m0001_tx_bucket_columns`), a tabela `budget_buckets` (`m0005`) e a tabela `bucket_rules`
  (`m0007`). F1.1 faz o **SEED** dos 6 potes, escreve a **lógica** (mapa + similaridade), liga na
  **ingestão** e expõe os **endpoints**.
- O ponto de ingestão a estender é onde `Categorizer().apply_to_database(db)` já roda hoje:
  `src/web/app.py:104` (sync de 1 item), `src/web/app.py:147` (`_sync_all_items`) e `src/cli.py:78`
  (comando `categorize`).

**Objetivo de F1.1:** após esta spec, toda transação importada recebe um `bucket_id` sugerido
automaticamente (mapa default + regras aprendidas), o usuário pode sobrescrever pela API/UI, e a
sobrescrita manual **aprende uma regra** e **propaga** para similares ainda sem meta — sem nunca
sobrescrever uma escolha manual existente.

> **Divergência apontada com o PRIMER:** o primer §4.3 sugere as cores dos potes apenas por nome
> ("verde, azul-aço, ciano, roxo, laranja, vermelho-vinho") e diz que o previsto default fica "0 a definir
> em F2.6". Esta spec **fixa valores hex concretos** (§4.2) e propõe um **seed de `planned_*` em `percent`
> somando 100%** (§4.2), para que F2.3 (Orçamento) tenha barras de "previsto" não-zeradas no dia 1 —
> mantendo `is_system=1` e tudo editável em F2.6. Se F2.6 preferir começar com tudo zerado, basta trocar
> os `planned_value` do seed (idempotência por `key`, não por valor — ver §4.1).

## 2. Escopo / fora de escopo

### 2.1 No escopo (F1.1)

  `POST /api/transactions/{id}/bucket` (set + propagação opcional), e (opcional)

## 3. Os 6 potes (modelo)

A tabela `budget_buckets` já existe (F0.1 `m0005`):

```sql
budget_buckets(
  id INTEGER PK,
  key TEXT UNIQUE NOT NULL,           -- estável, usado em código/seed/idempotência
  name TEXT NOT NULL,                 -- rótulo PT exibido na UI
  color TEXT,                         -- cor hex p/ a bolinha do BucketSelect
  planned_kind TEXT,                  -- 'percent' | 'amount'
  planned_value REAL,                 -- % da renda (se percent) OU valor fixo (se amount)
  sort_order INTEGER NOT NULL DEFAULT 0,
  is_system INTEGER NOT NULL DEFAULT 1
)
```

`transactions.bucket_id` (FK lógica → `budget_buckets.id`) é a "Meta". `transactions.bucket_source`
(`'auto' | 'manual' | 'rule'`) registra **como** a Meta foi atribuída — é o que sustenta a precedência
(§4.5). "Sem meta" = `bucket_id IS NULL`.

## 4. Backend

### 4.1 Seed dos 6 potes (idempotente)

**Onde:** função `seed_buckets(db)` em `src/web/repositories/buckets_repo.py`, chamada **uma vez** na
factory `create_app()` (após `apply_migrations`, junto do bootstrap) e exposta também via CLI
(`python -m src.cli seed-buckets`) e implicitamente pelo `POST /api/buckets/reclassify`.

**Idempotência:** o seed é `INSERT ... ON CONFLICT(key) DO NOTHING` (a coluna `key` é `UNIQUE`). Rodar N
vezes **não duplica** e **não sobrescreve** edições do usuário em `name/color/planned_*` (importante para
F2.6: depois que o usuário ajustar o previsto, um novo boot não reverte). Só insere o que faltar (cobre o
caso de adicionarmos um 7º pote no futuro sem mexer nos existentes).

```python
# src/web/repositories/buckets_repo.py
from typing import Any
from ...storage import Database

# Seed canônico dos 6 potes. 'key' é o identificador estável (idempotência).
# planned_kind='percent' / planned_value=% da renda — soma = 100 (ver nota §1).
BUCKET_SEED: list[dict[str, Any]] = [
    {"key": "liberdade_financeira", "name": "Liberdade Financeira", "color": "#22C55E",
     "planned_kind": "percent", "planned_value": 10.0, "sort_order": 1},
    {"key": "custos_fixos",        "name": "Custos Fixos",         "color": "#3B82F6",
     "planned_kind": "percent", "planned_value": 55.0, "sort_order": 2},
    {"key": "conforto",            "name": "Conforto",             "color": "#06B6D4",
     "planned_kind": "percent", "planned_value": 10.0, "sort_order": 3},
    {"key": "metas",               "name": "Metas",                "color": "#A855F7",
     "planned_kind": "percent", "planned_value": 10.0, "sort_order": 4},
    {"key": "prazeres",            "name": "Prazeres",             "color": "#F97316",
     "planned_kind": "percent", "planned_value": 10.0, "sort_order": 5},
    {"key": "conhecimento",        "name": "Conhecimento",         "color": "#9F1239",
     "planned_kind": "percent", "planned_value": 5.0,  "sort_order": 6},
]

def seed_buckets(db: Database) -> int:
    """Insere os 6 potes que faltarem. Idempotente (ON CONFLICT(key) DO NOTHING).
    Retorna quantas linhas foram inseridas nesta chamada."""
    inserted = 0
    with db._cursor() as cur:  # type: ignore[attr-defined]
        for b in BUCKET_SEED:
            cur.execute(
                """
                INSERT INTO budget_buckets
                  (key, name, color, planned_kind, planned_value, sort_order, is_system)
                VALUES (:key, :name, :color, :planned_kind, :planned_value, :sort_order, 1)
                ON CONFLICT(key) DO NOTHING
                """,
                b,
            )
            inserted += cur.rowcount if cur.rowcount > 0 else 0
    return inserted
```

| 2 | `custos_fixos` | Custos Fixos | `#3B82F6` (azul-aço) | percent | 55.0 |
| 3 | `conforto` | Conforto | `#06B6D4` (ciano) | percent | 10.0 |
| 4 | `metas` | Metas | `#A855F7` (roxo) | percent | 10.0 |
| 5 | `prazeres` | Prazeres | `#F97316` (laranja) | percent | 10.0 |
| 6 | `conhecimento` | Conhecimento | `#9F1239` (vermelho-vinho) | percent | 5.0 |

### 4.3 Atribuição de pote na ingestão (`apply_buckets_to_database`)

Novo orquestrador (em `src/agent/buckets.py`, ao lado do mapa, para ficar perto do domínio e testável
sem FastAPI). Espelha a forma de `Categorizer.apply_to_database` (mesmo padrão `db._cursor()`), mas para
`bucket_id`. **Roda depois** da categorização (precisa de `category` preenchido para o mapa funcionar).

Ordem de atribuição por transação (precedência §4.5):
1. Se `bucket_source='manual'` → **nunca** toca.
2. Se existe regra aprendida (`bucket_rules`) cujo `match_key` casa → set `bucket_id` da regra, `bucket_source='rule'`.
3. Senão, se o mapa default dá um pote para a `category` → set `bucket_id`, `bucket_source='auto'`.
4. Senão, deixa `NULL` ("Sem meta").

Para não sobrescrever uma sugestão automática "boa" por uma "fraca" em re-runs, a regra é: `manual`
nunca muda; `rule` só é (re)escrita por regra; `auto` pode ser **promovido** a `rule` (regra é mais
forte que mapa), mas `rule` **não** rebaixa para `auto`.

```python
# src/agent/buckets.py  (continuação)
from ..storage import Database
from .context import _merchant_key  # MESMA normalização de comerciante (context.py:72)

def _signed_key(merchant_key: str, amount: float) -> str:
    """match_key = '<sinal><merchant_key>', sinal '+' (entrada) ou '-' (saída)."""
    sign = "+" if amount >= 0 else "-"
    return f"{sign}{merchant_key}"

def apply_buckets_to_database(db: Database) -> dict[str, int]:
    """Pré-preenche transactions.bucket_id a partir de (1) regras aprendidas e
    (2) mapa default categoria→pote. NUNCA sobrescreve bucket_source='manual'.
    Precedência: manual > rule > auto > NULL (§4.5). Idempotente."""
    stats = {"by_rule": 0, "by_map": 0, "unmatched": 0, "skipped_manual": 0}

    # key -> id dos potes (uma query); regras: match_key -> bucket_id (uma query).
    buckets = {b["key"]: b["id"] for b in list_buckets(db)}
    rules = {r["match_key"]: r["bucket_id"] for r in list_rules(db)}

    with db._cursor() as cur:  # type: ignore[attr-defined]
        cur.execute(
            "SELECT id, amount, category, raw_description, description, "
            "bucket_id, bucket_source FROM transactions"
        )
        rows = cur.fetchall()
        for row in rows:
            if row["bucket_source"] == "manual":
                stats["skipped_manual"] += 1
                continue

            mkey = _signed_key(
                _merchant_key(row["raw_description"] or row["description"] or ""),
                float(row["amount"]),
            )
            target_id: int | None = None
            source: str | None = None

            if mkey in rules and rules[mkey] is not None:
                target_id, source = rules[mkey], "rule"
            else:
                bkey = classify_bucket(row["category"])
                if bkey and bkey in buckets:
                    target_id, source = buckets[bkey], "auto"

            if target_id is None:
                if row["bucket_id"] is None:
                    stats["unmatched"] += 1
                continue

            # não rebaixar 'rule' para 'auto'
            if row["bucket_source"] == "rule" and source == "auto":
                continue
            if row["bucket_id"] == target_id and row["bucket_source"] == source:
                continue  # nada a fazer (idempotência)

            cur.execute(
                "UPDATE transactions SET bucket_id=?, bucket_source=? WHERE id=?",
                (target_id, source, row["id"]),
            )
            stats["by_rule" if source == "rule" else "by_map"] += 1
    return stats
```

### 4.4 Auto-sugestão por similaridade (algoritmo)

**Gatilho:** o usuário define a Meta de uma transação manualmente — via
`POST /api/transactions/{id}/bucket` com `apply_to_similar=true` (ou via `PATCH` + a UI chamando o
endpoint de propagação). O efeito é triplo: (a) seta a transação alvo como `manual`; (b) aprende/atualiza
uma regra em `bucket_rules`; (c) propaga para similares **ainda sem meta**.

**O que é "mesmo tipo" (similaridade):** duas transações são do mesmo tipo quando têm o **mesmo
`match_key`**, onde:

```
match_key = sinal + " " ... = _signed_key(_merchant_key(raw_description ou description), amount)
          = ("+"|"-") + _merchant_key(...)
```

- **`_merchant_key`** é `src/agent/context.py:72` — `anonymize(raw).lower()` → tokens via `_TOKEN`
  (`[a-zà-ÿ0-9]+`), descarta `_NOISE_TOKENS` (`pessoa, num, conta, cpf, cnpj, compra, debito, credito,
  pagamento, de, da, do, no, na, boleto`) e dígitos, junta os **4 primeiros tokens**. Ex.: `"IFOOD *REST
  BOM PRATO 12/06"` → `"ifood rest bom prato"`.
- **Sinal**: `'+'` para `amount >= 0` (entrada/estorno), `'-'` para `amount < 0` (saída). Garante que
  uma **compra** no "Mercado X" e um **estorno** do "Mercado X" não compartilhem regra (são "tipos"
  diferentes para fins de orçamento). Atende ao requisito "mesmo sinal".

**Formato do `match_key` (decisão):** string `"<sinal><merchant_key>"`, ex. `"-ifood rest bom prato"`,
`"+nubank salario"`. É o que vai em `bucket_rules.match_key` (coluna `UNIQUE`). Curto, determinístico,
human-readable (ajuda debug), e a chave de comerciante já é anonimizada (sem PII).

**Algoritmo passo a passo** (em `transactions_repo.set_bucket(... apply_to_similar=True)`):

```
ENTRADA: tx_id, bucket_id (ou None p/ "Sem meta"), apply_to_similar: bool

1. tx = get_transaction(tx_id)            # 404 se não existe
   SE bucket_id não None: valida bucket_exists(bucket_id)  # 422 se inválido

2. # (a) seta a transação alvo como escolha do usuário
   UPDATE transactions SET bucket_id=:bucket_id, bucket_source='manual' WHERE id=:tx_id
   # 'Sem meta' (bucket_id=None) também é 'manual': é uma decisão explícita do usuário
   #  -> impede que o auto-mapa volte a preencher depois (§7 "Sem meta pega").

3. SE NÃO apply_to_similar: RETORNA {updated: 1, similar_affected: 0, similar_ids: []}

4. mkey = _signed_key(_merchant_key(tx.raw_description ou tx.description), tx.amount)
   SE mkey vazio ("" sem tokens úteis): RETORNA {updated:1, similar_affected:0, ...}
       (sem chave estável não há como propagar com segurança — ver §7)

5. # (b) aprende/atualiza a regra
   SE bucket_id não None:
       upsert_rule(match_key=mkey, bucket_id=bucket_id)   # ON CONFLICT(match_key) DO UPDATE
   SENÃO:
       delete_rule(match_key=mkey)        # "Sem meta" não cria regra; remove se existia

6. # (c) propaga para SIMILARES AINDA SEM META, recomputando o match_key linha a linha
   #  (não dá p/ filtrar por match_key no SQL — é derivado em Python via _merchant_key).
   alvos = []
   PARA cada row EM SELECT id, amount, raw_description, description
                      FROM transactions
                      WHERE bucket_id IS NULL            # só "Sem meta"
                        AND bucket_source IS NOT 'manual' # nunca um 'Sem meta' manual
                        AND id != tx_id:
       SE _signed_key(_merchant_key(row...), row.amount) == mkey:
           alvos.append(row.id)
   SE bucket_id não None E alvos:
       UPDATE transactions SET bucket_id=:bucket_id, bucket_source='rule'
        WHERE id IN (alvos)
   RETORNA {updated:1, similar_affected: len(alvos), similar_ids: alvos}
```

**Tratamento de conflito:**
- **Manual vence sempre.** O passo 6 filtra `bucket_id IS NULL AND bucket_source != 'manual'`, então uma
  transação que o usuário já marcou (inclusive como "Sem meta" manual) **nunca** é sobrescrita pela
  propagação. Idem para uma que já tem qualquer `bucket_id` (não é `NULL`).
- **Regra vs regra:** `upsert_rule` com `ON CONFLICT(match_key) DO UPDATE SET bucket_id=excluded.bucket_id`
  — a **última** escolha manual para aquele comerciante/sinal ganha (mais recente = mais correta).
- **Regra vs mapa:** na próxima ingestão, `apply_buckets_to_database` prioriza regra sobre mapa (§4.3),
  então a regra aprendida "fixa" a decisão do usuário para futuras importações do mesmo comerciante.

**Performance:** o passo 6 escaneia `transactions` recomputando `_merchant_key` por linha. Para a base
pessoal/familiar do deon-fin (ordem de 10³–10⁴ linhas) isso é trivial (< dezenas de ms) e roda **fora**
do request crítico de leitura. Mitigações documentadas (não obrigatórias agora): (i) restringir o scan a
`bucket_id IS NULL` (já feito — corta a maioria), (ii) opcionalmente persistir `merchant_key` materializado
numa coluna/índice em spec futura se a base crescer (registrado em §7 "performance").

### 4.5 Precedência (fonte da verdade)

> **manual > regra aprendida (`rule`) > mapa default (`auto`) > nulo ("Sem meta")**

| `bucket_source` atual | Pode ser sobrescrito por… | Não pode por… |
|---|---|---|
| `manual` | nada (só outra ação manual do usuário) | rule, auto, propagação |
| `rule` | manual; outra rule (upsert do mesmo match_key) | auto (não rebaixa) |
| `auto` | manual; rule (promoção) | — |
| `NULL` (sem meta, source nulo) | manual; rule (propagação); auto (mapa) | — |
| `NULL` + `manual` ("Sem meta" explícito) | só manual | rule, auto, propagação |

Implementado em dois lugares coerentes: `apply_buckets_to_database` (ingestão, §4.3) e
`set_bucket(...apply_to_similar)` (propagação, §4.4). Ambos checam `bucket_source` antes de escrever.

### 4.8 `bucket_source='manual'` na escrita parcial

Detalhe de implementação no router do PATCH (alinha com `transactions_repo.update_transaction`, F0.1
§4.2, que usa sentinela `...` para distinguir "não mexer" de "set NULL"): quando o body do PATCH **contém**
a chave `bucket_id` (mesmo que `null`), o router passa `bucket_source='manual'` ao repo; quando **não**
contém, `bucket_source` fica como sentinela (não muda). Assim `bucket_id: null` via PATCH = "Sem meta"
**manual** (decisão explícita), coerente com o `POST /{id}/bucket` (§4.4 passo 2).

### 6.3 UX da propagação ("aplicar às similares")

Fluxo recomendado quando o `BucketSelect` é usado **inline na tabela de Transações** (F2.2):

1. Usuário abre o dropdown, (opcional) digita em "Buscar meta…", clica num pote (ou "Sem meta").
2. **Update otimista**: a linha já mostra a nova Meta; dispara a mutation.
   - Caminho A (simples): chama `onChange(bucketId)` → a página faz `PATCH /api/transactions/{id}` com
     `{bucket_id}` (marca `manual`, sem propagar).
   - Caminho B (com propagação): chama `onChangeWithPropagation(bucketId, applyToSimilar)`.
3. **Confirmação de propagação** (Caminho B): após o set, a página chama
   `POST /api/transactions/{id}/bucket` com `apply_to_similar:false` **primeiro** para descobrir
   `similar_affected`? Não — para evitar 2 chamadas, o padrão é: chamar direto com `apply_to_similar:true`
   **somente após** o usuário confirmar num **toast/confirm**: _"Aplicar 'Conforto' a N transações
   parecidas?"_ com botões **Aplicar** / **Só esta**. Como N é conhecido apenas após a chamada, a UX
   adotada é:
   - **Aplicar imediato + desfazer:** chama com `apply_to_similar:true`, e a resposta
     (`similar_affected`, `similar_ids`) alimenta um **toast** _"Meta aplicada a N transações parecidas.
     [Desfazer]"_. **Desfazer** reverte as `similar_ids` para "Sem meta" (`PATCH bucket_id:null` em lote,
     ou um endpoint de undo em spec futura) e remove a regra. Esse padrão (otimista + undo) é o mais
     fluido para edição inline e é o recomendado por esta spec.
   - Alternativa conservadora (se preferir confirmar antes): primeira chamada `apply_to_similar:false`
     só seta a alvo e retorna `match_key`; um `GET` de contagem de similares (endpoint extra, fora do
     escopo) informaria N; então confirma. **Não** adotada aqui para manter 1 round-trip.
4. Em erro da mutation: rollback do update otimista + toast de erro (usa `ApiError.message` do `api.ts`).

> **Em Orçamento (F2.3)** o `BucketSelect` aparece para reclassificar uma linha de gasto; lá a propagação
> é menos comum — F2.3 pode usar só `onChange` (Caminho A). F1.1 entrega o componente capaz dos dois.

## 7. Regras de negócio / edge cases

- **Sinal no `match_key`.** Compra (`amount<0`) e estorno/entrada (`amount>=0`) do mesmo comerciante são
  **tipos diferentes** (prefixo `'-'` vs `'+'`). Regra e propagação respeitam o sinal (§4.4). Em **cartão**
  a convenção é invertida (compra positiva), mas o `match_key` usa o **sinal do `amount` cru** — o que
  importa para o orçamento é separar "saída de dinheiro" de "entrada"; como o cartão é tratado por conta
  de crédito, manter o sinal cru é consistente e simples. (Se no futuro quisermos normalizar pelo
  "sentido de consumo", troca-se `amount` por `spending_value` do `context.py` — registrado como melhoria,
  fora do escopo.)
- **"Sem meta".** `bucket_id IS NULL`. Selecionar "Sem meta" no `BucketSelect` é uma **decisão manual**:
  seta `bucket_source='manual'` e `bucket_id=NULL`, e **remove** a regra daquele `match_key` (não cria).
  Isso impede que o mapa default ou a propagação voltem a preencher aquela transação ("Sem meta pega").
- **Nunca sobrescrever manual.** Tanto `apply_buckets_to_database` quanto a propagação filtram
  `bucket_source != 'manual'` / `bucket_id IS NULL` (§4.3/§4.4). Coberto por teste (§10).
- **Transferências, pagamento de fatura, renda, custo financeiro NÃO entram em pote.** O mapa retorna
  `None` para `Transferências - PIX/TED`, `Credit card payment`, `Renda - Salário`, `Tarifas Bancárias`
  etc. (§4.2). Elas ficam "Sem meta" por design — não são consumo a planejar (coerente com
  `NON_SPENDING_CATEGORIES` / `CARD_PAYMENT_CATEGORIES` / `FINANCIAL_COST_CATEGORIES` de `context.py`).
  Se o usuário **forçar** uma Meta nelas manualmente, é permitido (decisão dele), mas a propagação não as
  alcança por padrão porque a maioria tem `match_key` "ruidoso" (PIX/TED são filtrados como
  `_is_transfer_noise` no contexto; aqui não bloqueamos o set manual, só não sugerimos).
- **`match_key` vazio.** Descrições que viram `""` após `_merchant_key` (só dígitos/ruído) **não** geram
  regra nem propagam (§4.4 passo 4) — sem chave estável, propagar seria perigoso (casaria tudo com `"-"`).
- **Pote inexistente no PATCH/POST.** `bucket_exists` → `422 validation_error` (§4.7).
- **Seed não rodou / pote removido.** `apply_buckets_to_database` resolve `key→id`; se o pote não existe,
  trata como não mapeado (não atribui) em vez de quebrar. O seed roda no boot, então em operação normal
  os 6 existem.
- **Idempotência.** Seed (`ON CONFLICT(key) DO NOTHING`), atribuição (compara estado antes de `UPDATE`) e
  `reclassify` convergem ao rodar N vezes.
- **Performance da propagação.** Scan de `transactions WHERE bucket_id IS NULL` recomputando
  `_merchant_key` por linha. Aceitável na escala pessoal (10³–10⁴). Roda em background no sync e sob
  demanda no `POST /{id}/bucket`. Melhoria futura registrada: materializar `merchant_key` numa coluna
  indexada se a base crescer (não nesta spec).
- **Concorrência.** `Database` é 1 conexão por request (F0.1/primer §2.1, `check_same_thread=False`); o
  `POST /{id}/bucket` faz set+propagação numa transação SQLite (`db._cursor()` commit/rollback). Sem
  multiusuário (primer §5.5), o risco de corrida é desprezível.

## 9. Critérios de aceite (verificáveis)

1. **Seed:** num banco novo, após subir o app, `GET /api/buckets` retorna **exatamente 6** potes com as
   `key` `liberdade_financeira, custos_fixos, conforto, metas, prazeres, conhecimento`, ordenados por
   `sort_order`, com `color` hex e `planned_kind='percent'`. Rodar o seed de novo **não** cria duplicatas
   e **não** reverte um `planned_value` editado.
2. **Mapa default:** após sync/`categorize`, uma transação categorizada como `Alimentação - Mercado`
   recebe `bucket_id` do pote **Conforto** com `bucket_source='auto'`; `Educação` → **Conhecimento**;
   `Investimentos` → **Liberdade Financeira**; `Moradia - Aluguel` → **Custos Fixos**;
   `Transferências - PIX` permanece **Sem meta** (`bucket_id IS NULL`).
3. **Manual via PATCH:** `PATCH /api/transactions/{id}` com `{ "bucket_id": 3 }` grava `bucket_id=3` e
   `bucket_source='manual'`; `{ "bucket_id": null }` deixa "Sem meta" manual. `bucket_id` inexistente →
   `422 validation_error` no envelope `{ error: { code, message } }`. Id inexistente → `404 not_found`.
4. **Propagação:** dado 1 transação "IFOOD ..." marcada via `POST /api/transactions/{id}/bucket` com
   `{ "bucket_id": 4, "apply_to_similar": true }`, todas as outras "IFOOD ..." com o **mesmo sinal** e
   **`bucket_id IS NULL` não-manual** passam a `bucket_id=4`, `bucket_source='rule'`; a resposta traz
   `similar_affected = N` e `similar_ids`. Transações já categorizadas (qualquer `bucket_id` não nulo) e
   as marcadas como `manual` **não** mudam.
5. **Sinal:** uma entrada (`amount>=0`) do mesmo comerciante **não** é afetada pela propagação de uma
   saída (`amount<0`) e vice-versa.
6. **"Sem meta" remove regra:** `POST /{id}/bucket` com `bucket_id:null` remove a regra do `match_key` (se
   existia) e não propaga.
7. **Precedência:** após criar uma regra e rodar `POST /api/buckets/reclassify`, transações do mesmo
   `match_key` que estavam `auto` viram `rule`; nenhuma `manual` muda.
8. **Idempotência geral:** `POST /api/buckets/reclassify` rodado 2× seguidas produz o mesmo estado (a
   segunda execução não altera contagens de `bucket_id` atribuídos).
9. **`budget.py` intacto:** `summarize_5030`/`BUDGET_MAP` continuam funcionando; testes de `test_budget.py`
   passam sem alteração.
10. **Frontend:** `BucketSelect` lista "Buscar meta…", "Sem meta" + 6 potes com bolinha de cor; o
    selecionado mostra ✓; escolher dispara a mutation; o toast/undo de "N transações parecidas" aparece
    quando `similar_affected > 0`.
