## 1. Objetivo

Levar a experiência de manutenção financeira para o frontend novo, mantendo a lógica atual do backend. A tela
deve ser operacional e direta: mostrar saúde dos dados, pendências de classificação, alertas de consistência e
ações seguras que hoje dependem do fluxo legado.

## 2. Escopo funcional

- Criar rota `web/app/(app)/manutencao/page.tsx` e item no menu principal.
- Consumir `GET /api/maintenance` com TanStack Query.
- Exibir blocos para:
  - saúde geral dos dados;
  - pendências de categorização/potes/tags quando o endpoint informar;
  - itens que exigem revisão manual;
  - mensagens de erro ou estado vazio.
- Preservar o shape atual do endpoint; se alguma informação estiver ausente, a UI deve degradar para estado vazio sem quebrar.
- Não executar ações destrutivas nesta versão. Se houver ações futuras, devem exigir confirmação explícita e teste próprio.

## 3. Arquitetura esperada

- Backend: preferir reusar o endpoint existente. Só criar repository/router novo se o shape atual não for suficiente para a UI.
- Frontend:
  - `web/hooks/useMaintenance.ts`
  - `web/components/manutencao/*`
  - testes em `web/tests/maintenance.test.ts`
- Layout: usar os mesmos componentes das telas já entregues (`Header`, `SectionCard`, `KpiCard`, `EmptyState`, `Skeleton`, `DataTable`).

## 4. Critérios de aceite

- `/manutencao` renderiza no Next com loading, erro, vazio e dados reais.
- O menu principal inclui Manutenção sem quebrar navegação mobile/desktop.
- `/api/maintenance` continua respondendo para consumidores legados.
- A tela funciona em produção same-origin com chamadas relativas para `/api`.
- Testes backend/frontend passam, e `npm run build` continua verde.

## 5. Testes mínimos

- Backend: regressão de `/api/maintenance` para garantir status 200 e shape estável.
- Frontend: renderização dos principais blocos, empty state e erro de API.
- E2E/smoke manual após F3.1: abrir `/manutencao` na VPS e validar que não há erro de hydration nem chamada para host dev.
