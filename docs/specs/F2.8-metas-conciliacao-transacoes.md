## 1. Conceito

Hoje a meta de poupança tem **Guardado** manual (`savings_goals.saved_amount`). O usuário quer **conciliar**:
apontar quais **transações** representam aportes àquela meta, e o **Guardado** passa a somar esses lançamentos
automaticamente (e o Progresso/"Por mês"/"Cabe na sobra" evoluem). É o mesmo padrão de **conciliação de ERP**:
casar lançamentos do extrato com um objetivo/conta.

**Princípios:**
- Uma transação pode estar vinculada a **no máximo uma** meta de poupança (`transactions.savings_goal_id`).
- O vínculo **não muda** a categorização da transação (continua com seu `bucket_id`/`tag`/competência) — só
  alimenta o progresso da meta. Sem dupla contagem no Orçamento.
- **Guardado efetivo** = baseline manual + soma dos lançamentos vinculados (ver §3).

## 2. Migration (m0021 — aditiva, idempotente)
```sql
ALTER TABLE transactions ADD COLUMN savings_goal_id INTEGER;   -- guard por PRAGMA table_info; FK lógica → savings_goals(id)
CREATE INDEX IF NOT EXISTS idx_tx_savings_goal ON transactions(savings_goal_id);
```
> FK lógica (sem `REFERENCES` rígido, padrão do projeto). Ao **excluir uma meta**, limpar o vínculo:
> `UPDATE transactions SET savings_goal_id=NULL WHERE savings_goal_id=?` dentro de `delete_goal` (savings_repo).

## 3. Cálculo do "Guardado" (derivado)
Em `savings_repo` (`_row_to_public`/`list_with_summary`/`get_goal`):
- `saved_manual = savings_goals.saved_amount` (baseline existente; pode ficar 0).
- `saved_from_tx = Σ contribuição` das transações com `savings_goal_id = meta` e **não** `hidden`.
  - **Contribuição** (decisão v1): `abs(amount)` do lançamento vinculado (um aporte/transferência conta
    positivo). Lançamentos que sejam **resgate** da meta podem ser marcados como saída → subtraem (campo
    opcional `direction` no link; v1 default = entrada). Documentar a convenção.
- **`saved_total = saved_manual + saved_from_tx`** → usar `saved_total` em **Guardado**, **Progresso**
  (`saved_total/target`), **Por mês** (`(target − saved_total)/term_months`) e **Cabe na sobra**
  (`summarize_wishlist` recebe `guardado = saved_total`).
- Expor nos payloads: `saved_manual`, `saved_from_tx`, `saved_total`, `linked_count`.

> Reusa `summarize_wishlist` (budget.py) — só troca a fonte do `guardado` por `saved_total`.

## 4. Endpoints
**Atribuição inline (reusa F2.2):**
- `PATCH /api/transactions/{id}` `{ savings_goal_id }` (parcial; `null` desvincula). Já é o padrão de mutação de
  transação — apenas **adicionar o campo** ao update e à serialização (`savings_goal_id`, e opcional
  `savings_goal_name` no GET de transações).

**Conciliação (no domínio savings):**
- `GET /api/savings-goals/{id}/transactions` → lançamentos **vinculados** `[{id, description, date, amount, account, bucket, tag}]` + `saved_from_tx`.
- `GET /api/savings-goals/{id}/candidates?month=&q=&type=&account_id=&bucket_id=&page=` → transações **não
  vinculadas** sugeridas para conciliar (filtros do F2.2). Default de sugestão: lançamentos do pote **"Metas"**
  e/ou transferências/aportes (heurística), mas permite ver todas.
- `POST /api/savings-goals/{id}/link` `{ transaction_ids:[...] }` → vincula em lote (seta `savings_goal_id`).
- `POST /api/savings-goals/{id}/unlink` `{ transaction_ids:[...] }` → desvincula (seta NULL).
- `GET /api/savings-goals` / `{id}` passam a trazer `saved_total/saved_from_tx/linked_count` (§3).
Erros no envelope F0.1; validar que a transação existe; vincular a uma meta troca de outra (move).

## 6. Regras / edge cases
- Uma transação só em **uma** meta; vincular a outra **move** (some das candidatas da primeira).
- Excluir meta → desvincula as transações (não apaga lançamentos).
- `hidden` (oculta dos relatórios) **não** conta no `saved_from_tx`.
- Convenção de sinal: v1 soma `abs(amount)`; lançamentos de **resgate** (opcional) subtraem — sinalizar na UI.
- Sem dupla contagem: o vínculo não altera bucket/competência; Orçamento/Painel seguem iguais.
- `saved_total` pode **passar do alvo** (progresso > 100%) → mostrar "Concluída" / 100%+ no card.
