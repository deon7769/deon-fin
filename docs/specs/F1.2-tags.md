## 1. Contexto & objetivo

O print mostra **dois** artefatos de Tags:

1. **Seletor de tag** (usado inline em cada transação): um dropdown com placeholder _"Selecione uma tag"_,
   um campo de busca _"Buscar tag…"_, uma ação **"+ Criar"** no topo (cria uma tag **on-the-fly** e já a
   seleciona), e a lista de tags como **pills coloridas**. As 7 tags-semente do print, com suas cores:
   **Alimentação** (amarelo), **Conforto** (vermelho), **Educação** (azul-claro), **Lazer** (vinho),
   **Saúde** (azul), **Transporte** (laranja), **Vestuário** (roxo) (primer §4.4).
2. **Página "Tags"** (rota `/tags`, item no menu lateral — grupo `MENU`, ícone `Tags`, já cadastrada em
   `F0.2` `nav.ts`): cabeçalho _"Aqui você pode criar e visualizar suas tags. As tags podem ser anexadas as
   transações"_, um contador _"**N tags encontradas**"_, uma tabela com **bolinha de cor + nome + ações
   editar/excluir**, e um botão amarelo **"+ Criar Tag"** que abre um modal (nome + seletor de cor).

Como isso se liga ao código atual:

- **Tag** é um **rótulo livre colorido** (primer §8, glossário), distinto de **Categoria**
  (`transactions.category`, granular, do `categorizer`/Pluggy) e de **Meta**/pote
  (`transactions.bucket_id`, F1.1). As três camadas coexistem por transação.
- O schema já foi **criado vazio em F0.1**: a tabela `tags(id, name UNIQUE, color, created_at)` (migration
  `m0006_tags`) e a coluna `transactions.tag_id` (migration `m0002_tx_tag_column`). **F0.1 não semeou nada**
  (escopo F0.1 §2.2: "Seeds das 7 tags são de F1.2. Aqui só criamos as tabelas vazias"). **F1.2 faz o
  SEED, o CRUD (repositório + endpoints) e a UI.**
- A **camada de repositórios** (`src/web/repositories/`) e o **router** `src/web/routers/tags.py` já existem
  como esqueleto em F0.1 (F0.1 §4.1/§5.1: `tags_repo.list_tags`/`tag_exists` declarados como stub; o router
  `tags.py` nasce vazio ou com no máximo `GET /api/tags` retornando `[]`). F1.2 dá corpo.
- O **handler global de erros** (F0.1 §5.2) já normaliza tudo para `{ "error": { "code", "message" } }` —
  os routers só levantam `HTTPException(status, detail)`.
- No **frontend**, `web/components/ui/TagSelect.tsx` existe como **casca** (`<select>` nativo, props
  `value/options/onChange/disabled/placeholder`, F0.2 §3.7). F1.2 o reescreve como **dropdown custom** com
  busca + "+ Criar" + pills, **mantendo as props públicas compatíveis** e o `aria-label` (mesmo movimento
  que F1.1 fez com `BucketSelect`). O componente base `Pill` (F0.2 §3.7) já aceita `color` hex livre — é o
  que usamos para as pills coloridas; `DataTable<T>`, `EmptyState`, `SectionCard` são reusados na página.

**Objetivo de F1.2:** após esta spec, as 7 tags do print existem (seed idempotente), há CRUD completo de
tags via API (com nome único, validação de cor hex e regra de exclusão que **desvincula** sem apagar
transações), o `TagSelect` é um dropdown funcional com busca/criação on-the-fly/pills, e a página `/tags`
lista/cria/edita/exclui tags com estados de loading/vazio/erro.

> **Decisões/divergências com o material (resolvidas nesta spec):**
> - O primer §4.4 lista as 7 tags por nome+cor "por descrição" (amarelo, vermelho, azul-claro, vinho, azul,
>   laranja, roxo). **Esta spec fixa os hex concretos** (§4.1), alinhados ao design system do primer §7
>   (ex.: amarelo = `accent #F5B301`, vermelho = `negative #EF4444`) e ao tom dos pots de F1.1 para
>   coerência visual no dark theme.
> - O contador de uso por tag (`tx_count`) é **opcional** no `GET /api/tags`; **esta spec o inclui** (via
>   `LEFT JOIN`), porque a página `/tags` se beneficia de mostrar quantas transações usam cada tag e o
>   `DELETE` precisa reportar quantas foram desvinculadas. Custo desprezível na escala pessoal (§7).

## 2. Escopo / fora de escopo

### 2.1 No escopo (F1.2)

**Backend**
- **Seed idempotente** das 7 tags em `tags` (nome + cor hex do print), via `seed_tags(db)` no boot, espelhando
  o padrão de `seed_buckets` de F1.1 (`ON CONFLICT(name) DO NOTHING`).
- **Repositório** `tags_repo` com corpo: `list_tags` (com `tx_count`), `get_tag`, `tag_exists`, `create_tag`,
  `update_tag`, `delete_tag` (com a regra `UPDATE transactions SET tag_id=NULL WHERE tag_id=?` — **não apaga
  transações**), `seed_tags`, e helpers de validação (`name_taken`, normalização de cor).
- **Endpoints** (contratos completos com JSON e erros, seguindo primer §6 e o padrão de F0.1):
  - `GET /api/tags` → `{ items: [{ id, name, color, tx_count }] }`.
  - `POST /api/tags` `{ name, color }` → cria (nome único case-insensitive; cor hex válida; `409` se
    duplicado, `422` se cor/nome inválidos).
  - `PATCH /api/tags/{id}` `{ name?, color? }` → edita parcial (mesmas validações; `409` se renomear para um
    nome já usado; `404` se id inexistente).
  - `DELETE /api/tags/{id}` → remove a tag e **desvincula** das transações; retorna `untagged` (quantas
    transações ficaram com `tag_id=NULL`).
  - `PATCH /api/transactions/{id}` `{ tag_id }` (definir/limpar a tag de uma transação) — **já previsto** no
    padrão de mutação parcial do primer §6 e na assinatura `update_transaction` de F0.1 §4.2. F1.2 **apenas
    referencia** o efeito de `tag_id`; **não redefine** o endpoint inteiro (o corpo do PATCH genérico é F2.2).

**Frontend**
- **`TagSelect`** funcional (substitui a casca de F0.2): dropdown com botão/placeholder _"Selecione uma
  tag"_, popover com input _"Buscar tag…"_, ação **"+ Criar"** no topo (cria on-the-fly e já seleciona),
  lista de tags como `Pill` coloridas, estado vazio. Props, estados, acessibilidade. Usado inline na tabela
  de Transações (F2.2) e possivelmente em Faturas (F2.4).
- **Página `/tags`** (substitui o placeholder de F0.2 `app/(app)/tags/page.tsx`): cabeçalho com o texto do
  print + contador "N tags encontradas" + `DataTable` (bolinha de cor, nome, ações editar inline/excluir
  com confirmação) + botão "+ Criar Tag" (abre modal: nome + seletor de cor a partir de uma **paleta**,
  permitindo hex livre). Estados loading/vazio/erro.
- **Hooks TanStack Query**: `useTags` (query), `useCreateTag`/`useUpdateTag`/`useDeleteTag` (mutations com
  invalidação de cache), e `useSetTag` (mutation do `tag_id` de transação, análogo a `useSetBucket` de F1.1).

### 2.2 Fora de escopo (vão para outras specs)

- A **coluna/seletor de Tag dentro da tabela de Transações** (montagem da página, paginação, filtros, layout
  da linha) → **F2.2**. F1.2 entrega o `TagSelect` e os hooks; F2.2 os pluga na linha.
- O **corpo completo do `PATCH /api/transactions/{id}`** (todos os campos: `bucket_id`, `tag_id`, `hidden`,
  `note`, `reference_month`) → **F2.2** (F1.1 já especificou o efeito de `bucket_id`; F1.2 só o de `tag_id`).
- **Uso em Faturas** (Tag por lançamento de cartão) → **F2.4** (reusa `TagSelect`/`useSetTag`).
- **Filtrar transações por tag** (`?tag_id=` no `GET /api/transactions`) → **F2.2** (a assinatura já prevê
  `tag_id` em `transactions_repo.list_transactions`, F0.1 §4.2; F1.2 não monta a listagem de transações).
- **Multi-tag por transação** — **fora**: o modelo é **uma tag por transação** (`transactions.tag_id` é uma
  coluna escalar, primer §4.2/§8 "tag única exibida"). Se um dia virar N:N, é spec própria (§7).
- Mudar/ativar `PRAGMA foreign_keys` — F0.1 §3.4 mantém a FK `tag_id → tags(id)` como **lógica** (integridade
  garantida na camada de repositório). F1.2 segue a mesma decisão (o `DELETE` faz o `UPDATE ... tag_id=NULL`
  manualmente, justamente porque não há `ON DELETE SET NULL` ativo).

## 3. Modelo de dados (já criado em F0.1; F1.2 só usa/semeia)

A tabela `tags` já existe (F0.1 migration `m0006_tags`):

```sql
tags(
  id         INTEGER PRIMARY KEY,
  name       TEXT UNIQUE NOT NULL,                       -- rótulo exibido (único)
  color      TEXT,                                       -- cor hex p/ a bolinha/pill (ex.: '#F5B301')
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
)
```

E a coluna `transactions.tag_id` (F0.1 migration `m0002_tx_tag_column`):

```sql
-- em transactions: tag_id INTEGER   (FK lógica -> tags(id); NULL = sem tag)
```

- `transactions.tag_id` (FK lógica → `tags.id`) é a tag única da transação. "Sem tag" = `tag_id IS NULL`.
- **Unicidade do nome:** a coluna `name` é `UNIQUE` (constraint de banco, case-**sensitive** no SQLite por
  padrão). Esta spec adiciona unicidade **case-insensitive** na camada de aplicação (§7) para evitar
  "Lazer" e "lazer" coexistirem — o `UNIQUE` do banco é a rede de segurança; o repo valida antes com
  `LOWER(name)`.
- **Sem `bucket_source`/`tag_source`:** tag não tem auto-sugestão por similaridade nesta spec (diferente da
  Meta de F1.1). Tag é sempre uma escolha manual/explícita do usuário. (Auto-tagging por regra é melhoria
  futura registrada em §7, fora do escopo.)

> **Por que não reaproveitar a coluna `category`/pote?** Tag é ortogonal: o usuário pode marcar
> "Alimentação" (tag) numa transação cuja `category` é `Transporte - App` e cujo pote é `Custos Fixos`. As
> 7 tags-semente **coincidem em nome** com alguns conceitos de categoria/pote por conveniência do print, mas
> são entidades independentes (`tags` é tabela própria). Não há sincronização automática entre `tag` e
> `category`/`bucket`.

## 4. Backend

### 4.1 Seed das 7 tags (idempotente)

**Onde:** função `seed_tags(db)` em `src/web/repositories/tags_repo.py`, chamada **uma vez** no bootstrap da
factory `create_app()` (`src/web/app.py`), junto de `seed_buckets(db)` de F1.1 (ambos após
`apply_migrations`, que já roda no `Database.__init__`). Exposta também via CLI (`python -m src.cli seed-tags`).

**Idempotência:** `INSERT ... ON CONFLICT(name) DO NOTHING` (a coluna `name` é `UNIQUE`). Rodar N vezes **não
duplica** e **não sobrescreve** `color` editado pelo usuário (se ele recolorir "Saúde", um novo boot não
reverte). Só insere o que faltar (cobre o caso de o usuário ter excluído uma tag-semente — o seed a
recria; se isso for indesejável, ver nota abaixo). Espelha exatamente `seed_buckets` de F1.1 §4.1.

```python
# src/web/repositories/tags_repo.py
from typing import Any
from ...storage import Database

# Seed canônico das 7 tags do print (primer §4.4). 'name' é UNIQUE (idempotência).
# Cores hex alinhadas ao design system do primer §7 (amarelo=accent, vermelho=negative...).
TAG_SEED: list[dict[str, str]] = [
    {"name": "Alimentação", "color": "#F5B301"},   # amarelo  (== accent do primer §7)
    {"name": "Conforto",    "color": "#EF4444"},   # vermelho (== negative do primer §7)
    {"name": "Educação",    "color": "#38BDF8"},   # azul-claro
    {"name": "Lazer",       "color": "#9F1239"},   # vinho
    {"name": "Saúde",       "color": "#3B82F6"},   # azul
    {"name": "Transporte",  "color": "#F97316"},   # laranja
    {"name": "Vestuário",   "color": "#A855F7"},   # roxo
]

def seed_tags(db: Database) -> int:
    """Insere as 7 tags que faltarem. Idempotente (ON CONFLICT(name) DO NOTHING).
    Retorna quantas linhas foram inseridas nesta chamada."""
    inserted = 0
    with db._cursor() as cur:  # type: ignore[attr-defined]
        for t in TAG_SEED:
            cur.execute(
                "INSERT INTO tags (name, color) VALUES (:name, :color) "
                "ON CONFLICT(name) DO NOTHING",
                t,
            )
            inserted += cur.rowcount if cur.rowcount and cur.rowcount > 0 else 0
    return inserted
```

**Seed de nomes e cores (decisão desta spec):**

| name | cor (descrição no print) | color (hex) | origem do hex |
|---|---|---|---|
| Alimentação | amarelo | `#F5B301` | `accent` (primer §7) |
| Conforto | vermelho | `#EF4444` | `negative` (primer §7) |
| Educação | azul-claro | `#38BDF8` | sky-400 |
| Lazer | vinho | `#9F1239` | == `conhecimento` de F1.1 (vinho) |
| Saúde | azul | `#3B82F6` | == `custos_fixos`/`Saúde` de F1.1 (azul) |
| Transporte | laranja | `#F97316` | == `prazeres` de F1.1 (laranja) |
| Vestuário | roxo | `#A855F7` | == `metas` de F1.1 (roxo) |

> **Recriação após exclusão (edge case explícito):** como o seed insere "o que faltar por `name`", se o
> usuário **excluir** uma tag-semente e o app **reiniciar**, o seed a **recria**. Para a base pessoal isso é
> aceitável (e raro). Se for indesejável, a alternativa é o seed rodar **apenas quando a tabela `tags` está
> vazia** (`SELECT COUNT(*) FROM tags == 0`) — registrado como variante; **esta spec adota o `ON CONFLICT`
> por nome** (mais simples, idempotente, e garante que as 7 existam em operação normal), e o `seed-tags` do
> CLI permite re-semear sob demanda. F2.* não deve depender da existência fixa de uma tag-semente específica
> (tags são editáveis/removíveis pelo usuário).

### 4.2 Validação de cor e de nome

Helpers no `tags_repo` (ou em `src/web/repositories/_validators.py` se preferir compartilhar):

```python
import re

# Aceita #RGB e #RRGGBB (com ou sem '#'? -> exigimos '#'). Case-insensitive.
_HEX_RE = re.compile(r"^#(?:[0-9a-fA-F]{3}|[0-9a-fA-F]{6})$")

def normalize_color(color: str | None) -> str | None:
    """Valida e normaliza uma cor hex. Retorna a cor em minúsculas com '#'.
    Lança ValueError se inválida. None/'' -> None (tag sem cor é permitida)."""
    if color is None:
        return None
    c = color.strip()
    if c == "":
        return None
    if not _HEX_RE.match(c):
        raise ValueError("cor inválida (use hex #RGB ou #RRGGBB)")
    return c.lower()

def normalize_name(name: str) -> str:
    """Trim + colapsa espaços. Lança ValueError se vazio ou > 40 chars."""
    n = " ".join((name or "").split())
    if not n:
        raise ValueError("nome obrigatório")
    if len(n) > 40:
        raise ValueError("nome muito longo (máx. 40)")
    return n

def name_taken(db: Database, name: str, *, exclude_id: int | None = None) -> bool:
    """True se já existe uma tag com o mesmo nome (case-insensitive),
    ignorando exclude_id (usado no update para permitir 'salvar sem mudar o nome')."""
    with db._cursor() as cur:  # type: ignore[attr-defined]
        if exclude_id is None:
            cur.execute("SELECT 1 FROM tags WHERE LOWER(name)=LOWER(?) LIMIT 1", (name,))
        else:
            cur.execute(
                "SELECT 1 FROM tags WHERE LOWER(name)=LOWER(?) AND id<>? LIMIT 1",
                (name, exclude_id),
            )
        return cur.fetchone() is not None
```

**Regras de validação (fonte da verdade):**
- **Nome:** obrigatório, `trim` + colapso de espaços, **1–40 caracteres**, **único case-insensitive**
  (`name_taken`). Duplicado → **409 conflict**. Vazio/longo → **422 validation_error**.
- **Cor:** opcional; se presente, hex `#RGB` ou `#RRGGBB` (validado por `_HEX_RE`), normalizada para
  minúsculas com `#`. Inválida → **422 validation_error**. Ausente/`""`/`null` → `color = NULL` (tag sem
  cor; a UI usa uma cor neutra padrão — §6).

### 4.3 Repositório `tags_repo` (corpo em F1.2)

Em F0.1 `tags_repo` nasceu com stubs (`list_tags`, `tag_exists`). F1.2 dá corpo a tudo:

```python
# src/web/repositories/tags_repo.py  (além de TAG_SEED/seed_tags/validators acima)
from typing import Any, TypedDict
from ...storage import Database

class TagRow(TypedDict):
    id: int
    name: str
    color: str | None
    tx_count: int

def list_tags(db: Database) -> list[dict[str, Any]]:
    """Lista todas as tags com contagem de uso (tx_count via LEFT JOIN).
    Ordenadas por name (case-insensitive). Campos: id, name, color, tx_count."""
    with db._cursor() as cur:  # type: ignore[attr-defined]
        cur.execute(
            """
            SELECT t.id, t.name, t.color, COUNT(tx.id) AS tx_count
              FROM tags t
              LEFT JOIN transactions tx ON tx.tag_id = t.id
             GROUP BY t.id, t.name, t.color
             ORDER BY LOWER(t.name)
            """
        )
        return [dict(r) for r in cur.fetchall()]

def get_tag(db: Database, tag_id: int) -> dict[str, Any] | None:
    with db._cursor() as cur:  # type: ignore[attr-defined]
        cur.execute(
            """
            SELECT t.id, t.name, t.color, COUNT(tx.id) AS tx_count
              FROM tags t LEFT JOIN transactions tx ON tx.tag_id = t.id
             WHERE t.id = ?
             GROUP BY t.id, t.name, t.color
            """,
            (tag_id,),
        )
        row = cur.fetchone()
        return dict(row) if row else None

def tag_exists(db: Database, tag_id: int) -> bool:
    """Validação usada pelo PATCH /transactions (tag_id) — F0.1 §4.2."""
    with db._cursor() as cur:  # type: ignore[attr-defined]
        cur.execute("SELECT 1 FROM tags WHERE id=? LIMIT 1", (tag_id,))
        return cur.fetchone() is not None

def create_tag(db: Database, *, name: str, color: str | None) -> dict[str, Any]:
    """Cria uma tag. Pressupõe nome já normalizado/validado e cor já normalizada
    pelo router; faz a verificação de duplicidade (case-insensitive) e insere.
    Levanta ValueError('duplicate') se o nome já existe -> router converte em 409."""
    if name_taken(db, name):
        raise ValueError("duplicate")
    with db._cursor() as cur:  # type: ignore[attr-defined]
        cur.execute("INSERT INTO tags (name, color) VALUES (?, ?)", (name, color))
        new_id = cur.lastrowid
    return get_tag(db, int(new_id))  # type: ignore[arg-type]

def update_tag(
    db: Database, tag_id: int, *,
    name: str | None = None,        # já normalizado pelo router, ou None p/ "não mexer"
    color: str | None = ...,        # sentinela '...' = não mexer; None = limpar cor
) -> dict[str, Any] | None:
    """Atualiza name/color parcialmente. Retorna a tag atualizada, ou None se o id
    não existe (router -> 404). Levanta ValueError('duplicate') se o novo name colide."""
    existing = get_tag(db, tag_id)
    if existing is None:
        return None
    sets: list[str] = []
    params: list[Any] = []
    if name is not None:
        if name_taken(db, name, exclude_id=tag_id):
            raise ValueError("duplicate")
        sets.append("name=?"); params.append(name)
    if color is not ...:
        sets.append("color=?"); params.append(color)
    if sets:
        params.append(tag_id)
        with db._cursor() as cur:  # type: ignore[attr-defined]
            cur.execute(f"UPDATE tags SET {', '.join(sets)} WHERE id=?", params)
    return get_tag(db, tag_id)

def delete_tag(db: Database, tag_id: int) -> dict[str, Any] | None:
    """Remove a tag e DESVINCULA das transações (tag_id=NULL) — NÃO apaga transações.
    Retorna {deleted_id, untagged} (quantas transações foram desvinculadas), ou None
    se o id não existe (router -> 404). Faz tudo numa transação (db._cursor commita)."""
    existing = get_tag(db, tag_id)
    if existing is None:
        return None
    with db._cursor() as cur:  # type: ignore[attr-defined]
        cur.execute("UPDATE transactions SET tag_id=NULL WHERE tag_id=?", (tag_id,))
        untagged = cur.rowcount or 0
        cur.execute("DELETE FROM tags WHERE id=?", (tag_id,))
    return {"deleted_id": tag_id, "untagged": int(untagged)}
```

> **Regra de exclusão (requisito central da spec):** `delete_tag` faz, **na mesma transação**,
> `UPDATE transactions SET tag_id=NULL WHERE tag_id=?` **antes** do `DELETE FROM tags`. Isso garante que
> **nenhuma transação é apagada** — elas apenas voltam a "Sem tag". O `cur.rowcount` do `UPDATE` é a
> contagem de desvinculadas (`untagged`), devolvida ao cliente. Como o `db._cursor()` (db.py:115) commita ao
> sair sem exceção e faz rollback em erro, a operação é atômica.

### 4.4 Endpoints

Router: `src/web/routers/tags.py` (criado em F0.1; F1.2 dá corpo). Erros seguem o **handler global de F0.1**
(`{ "error": { "code", "message" } }`, F0.1 §5.2) — os routers só levantam `HTTPException(status, detail)`.

#### `POST /api/tags` — cria uma tag

- **Método/rota:** `POST /api/tags`
- **Body:** `{ "name": "Pets", "color": "#10B981" }` (cor opcional).
- **Validações (router):** `normalize_name(name)` (422 se vazio/>40); `normalize_color(color)` (422 se hex
  inválido); duplicidade case-insensitive em `create_tag` (409 se já existe).
- **201 Created:**

```json
{ "id": 8, "name": "Pets", "color": "#10b981", "tx_count": 0 }
```

- **Erros:**
  - `409 conflict` — nome já existe (case-insensitive):
    `{ "error": { "code": "conflict", "message": "já existe uma tag com esse nome" } }`.
  - `422 validation_error` — nome vazio/muito longo **ou** cor hex inválida:
    `{ "error": { "code": "validation_error", "message": "cor inválida (use hex #RGB ou #RRGGBB)" } }`.

```python
@router.post("/tags", status_code=201)
def create_tag_endpoint(body: TagCreate, db: Database = Depends(get_db)):
    try:
        name = normalize_name(body.name)
        color = normalize_color(body.color)
    except ValueError as e:
        raise HTTPException(422, str(e))
    try:
        return tags_repo.create_tag(db, name=name, color=color)
    except ValueError as e:
        if str(e) == "duplicate":
            raise HTTPException(409, "já existe uma tag com esse nome")
        raise HTTPException(422, str(e))
```

> **Status 201 vs 200:** criação retorna **201** (recurso criado), coerente com REST. O front (`useCreateTag`)
> trata 2xx; o `apiFetch` de F0.2 não distingue 200/201 (ambos `res.ok`). A criação **on-the-fly** do
> `TagSelect` usa o mesmo endpoint e já recebe o `{ id, ... }` para selecionar (§6.3).

#### `PATCH /api/tags/{id}` — edita nome e/ou cor

- **Método/rota:** `PATCH /api/tags/{id}`
- **Body (parcial):** `{ "name": "Alimentação e Bebidas" }` ou `{ "color": "#22C55E" }` ou ambos.
- **Semântica de `color`:** ver §4.5 (distinção "não mexer" × "limpar cor").
- **200:**

```json
{ "id": 1, "name": "Alimentação e Bebidas", "color": "#22c55e", "tx_count": 42 }
```

- **Erros:**
  - `404 not_found` — id inexistente: `{ "error": { "code": "not_found", "message": "tag não encontrada" } }`.
  - `409 conflict` — renomear para um nome já usado por outra tag.
  - `422 validation_error` — nome vazio/longo ou cor inválida.

```python
@router.patch("/tags/{tag_id}")
def update_tag_endpoint(tag_id: int, body: TagUpdate, db: Database = Depends(get_db)):
    fields = body.model_dump(exclude_unset=True)   # só o que veio no corpo
    name = None
    color: object = ...                            # sentinela "não mexer"
    try:
        if "name" in fields:
            name = normalize_name(body.name or "")
        if "color" in fields:
            color = normalize_color(body.color)    # None => limpar cor
    except ValueError as e:
        raise HTTPException(422, str(e))
    try:
        updated = tags_repo.update_tag(db, tag_id, name=name, color=color)  # type: ignore[arg-type]
    except ValueError as e:
        if str(e) == "duplicate":
            raise HTTPException(409, "já existe uma tag com esse nome")
        raise HTTPException(422, str(e))
    if updated is None:
        raise HTTPException(404, "tag não encontrada")
    return updated
```

#### `DELETE /api/tags/{id}` — remove a tag e desvincula das transações

- **Método/rota:** `DELETE /api/tags/{id}`
- **Body:** nenhum.
- **Efeito:** `UPDATE transactions SET tag_id=NULL WHERE tag_id=?` (desvincula, **não apaga transações**) e
  então `DELETE FROM tags WHERE id=?` — atômico (§4.3).
- **200:**

```json
{ "deleted_id": 6, "untagged": 28 }
```

  - `untagged` = quantas transações tinham essa tag e foram desvinculadas (voltaram a `tag_id=NULL`).
- **Erros:** `404 not_found` — id inexistente.

```python
@router.delete("/tags/{tag_id}")
def delete_tag_endpoint(tag_id: int, db: Database = Depends(get_db)):
    result = tags_repo.delete_tag(db, tag_id)
    if result is None:
        raise HTTPException(404, "tag não encontrada")
    return result
```

> **Sem confirmação no backend:** a confirmação ("Tem certeza?") é **só de UI** (§6.4). O endpoint exclui
> direto. Como nenhuma transação é apagada (só desvinculada) e o `untagged` é reportado, o usuário pode
> recriar a tag e re-marcar se errar. **Não** há "soft delete" nesta spec (registrado como possível melhoria
> em §7).

#### `PATCH /api/transactions/{id}` — define/limpa a tag de uma transação (referência)

> **Não redefinido aqui.** Este é o **mesmo** endpoint de mutação parcial do primer §6 / F0.1 §4.2
> (`update_transaction`), cujo corpo completo (todos os campos) é montado em **F2.2**. F1.1 já especificou o
> efeito de `bucket_id`; **F1.2 apenas especifica o efeito de `tag_id`**:

- **Body (parcial):** `{ "tag_id": 3 }` (definir) ou `{ "tag_id": null }` (limpar → "Sem tag").
- **Efeito:** seta `transactions.tag_id`. **Validação:** se `tag_id` não-nulo, o repo/handler chama
  `tags_repo.tag_exists(db, tag_id)`; se não existir → **422 validation_error** (`"tag_id inválido"`). Id de
  transação inexistente → **404 not_found**.
- **200 (trecho relevante):**

```json
{ "id": "a1b2c3...", "tag_id": 3, "bucket_id": 4, "bucket_source": "manual", "hidden": 0, "note": null }
```

- **Diferença para a Meta:** **não há propagação por similaridade** para `tag_id` (ao contrário de
  `bucket_id` em F1.1). Setar a tag afeta **só aquela transação**. Logo, **não** existe
  `POST /api/transactions/{id}/tag` (não há análogo do `POST /{id}/bucket`); o `PATCH` basta.

> **Coordenação com F2.2 (evitar conflito):** o `PATCH` é um endpoint único que aceita campos parciais.
> F1.1 definiu o efeito de `bucket_id` (marca `bucket_source='manual'`); F1.2 define o efeito de `tag_id`
> (set direto, valida `tag_exists`); F2.2 junta tudo no router final, garantindo que enviar
> `{ "tag_id": 3 }` e `{ "bucket_id": 4 }` no **mesmo** PATCH funcione (cada campo tratado pelo seu repo). A
> sentinela `...` de `update_transaction` (F0.1 §4.2) distingue "não mexer" de "set NULL" por campo —
> `tag_id` segue essa convenção: chave **ausente** = não mexer; `tag_id: null` = limpar.

### 4.5 "Não mexer" × "limpar cor" no PATCH de tag

`color` é opcional e pode ser explicitamente apagada. Para distinguir os dois casos no `PATCH /api/tags/{id}`:

- **Chave `color` ausente** no body → **não mexe** na cor (mantém a atual). No repo, `color=...` (sentinela).
- **`color: null`** ou **`color: ""`** no body → **limpa** a cor (`color = NULL`). `normalize_color` converte
  `""`→`None`; `null` chega como `None`. Como `body.model_dump(exclude_unset=True)` só inclui chaves
  presentes, o router detecta `"color" in fields` e passa `None` ao repo (que grava `NULL`).
- **`color: "#22C55E"`** → valida e grava `#22c55e`.

Mesma lógica para `name` (mas `name` nunca pode virar `NULL`/vazio — é `422`). Esta distinção espelha a
sentinela `...` de `update_transaction` (F0.1 §4.2) e o tratamento de `bucket_id` em F1.1 §4.8 — **padrão
consistente em todo o projeto**.

## 5. Reconciliação com F1.1 (consistência de padrão)

F1.2 e F1.1 são specs irmãs (Tag ↔ Meta). Para não divergir nem conflitar:

| Aspecto | F1.1 (Meta/pote) | F1.2 (Tag) | Consistente? |
|---|---|---|---|
| Tabela | `budget_buckets` (seed 6, `is_system`) | `tags` (seed 7) | sim (mesmo padrão de seed `ON CONFLICT`) |
| Coluna em `transactions` | `bucket_id` + `bucket_source` | `tag_id` (sem `_source`) | tag não precisa de fonte (sempre manual) |
| Seed | `seed_buckets(db)` no boot + CLI | `seed_tags(db)` no boot + CLI | mesmo lugar (`create_app` após migrations) |
| `GET` lista | `GET /api/buckets` → `{items}` | `GET /api/tags` → `{items}` | mesmo envelope |
| Mutação em tx | `PATCH {bucket_id}` + `POST /{id}/bucket` (propaga) | `PATCH {tag_id}` (sem propagação) | tag não propaga (§4.4) |
| Componente | `BucketSelect` (dropdown custom, "Buscar meta…", "Sem meta", ✓) | `TagSelect` (dropdown custom, "Buscar tag…", "+ Criar", pills) | UX análoga; tag adiciona "criar on-the-fly" |
| CRUD da entidade | **não** há (potes são fixos/sistema; editar `planned_*` é F2.6) | **sim** (criar/editar/excluir tags) | diferença de domínio: tags são do usuário |
| Hooks | `useBuckets`, `useSetBucket` | `useTags`, `useCreateTag/UpdateTag/DeleteTag`, `useSetTag` | mesma stack (TanStack) |
| Validação no PATCH de tx | `bucket_exists` → 422 | `tag_exists` → 422 | mesmo padrão |

**Diferenças intencionais (não são conflito):**
1. **Tag tem CRUD de entidade; pote não.** Potes são 6 fixos do método (editáveis só em `planned_*`, F2.6);
   tags são criadas/renomeadas/excluídas pelo usuário. Por isso F1.2 tem `POST/PATCH/DELETE /api/tags`, que
   não existem para buckets.
2. **Tag não propaga por similaridade.** Não há `bucket_rules` equivalente para tags nesta spec. Setar uma
   tag afeta só a transação. (Auto-tag por regra é melhoria futura — §7.)
3. **Tag tem "+ Criar" on-the-fly no seletor.** O `TagSelect` cria uma tag nova direto do dropdown (o
   `BucketSelect` não cria potes). Isso usa o `POST /api/tags` e invalida o cache de tags (§6.3).

**Não conflitar no `PATCH /api/transactions/{id}`:** F1.1, F1.2 e F2.2 tocam o mesmo endpoint. A regra é:
cada spec descreve **o efeito do seu campo**; F2.2 implementa o router final que aceita todos os campos
parciais simultaneamente. F1.2 **não** reescreve o handler — só documenta `tag_id` e a validação `tag_exists`.

| prop | tipo | req? | default | descrição |
|---|---|---|---|---|
| `value` | `number \| null` | não | `null` | `tag_id` selecionado; `null` mostra o placeholder |
| `options` | `Tag[]` | sim | — | tags de `/api/tags` (caller passa `useTags().data`) |

#### Estrutura visual / comportamento

- **Botão (trigger):** se `value` selecionado, mostra a `Pill` da tag (cor + nome); senão, o texto
  `placeholder` ("Selecione uma tag") em `text-muted`. Abre o popover ao clicar/Enter/Espaço.
- **Popover:**
  1. **Input de busca** _"Buscar tag…"_ no topo (autofoco ao abrir). Filtra `options` por `name`
     (case/acento-insensível, via `.normalize("NFD").replace(/\p{Diacritic}/gu, "")`).
  2. **Ação "+ Criar"** no topo (abaixo/junto da busca, como no print): visível **sempre** que `onCreate`
     existe; o rótulo vira **"+ Criar «{query}»"** quando há texto digitado que **não casa exatamente** com
     uma tag existente (cria com esse nome); com busca vazia, "+ Criar" abre um mini-input para o nome (ou
     usa o texto da busca). Ao acionar: chama `onCreate(name)`, recebe a `Tag`, **seleciona-a**
     (`onChange(tag.id)`) e fecha o popover.
  3. **Lista** de tags como `Pill` coloridas (`<Pill color={t.color}>{t.name}</Pill>`), uma por linha,
     clicáveis; a selecionada (`t.id === value`) recebe um **✓** e destaque (`bg-surface2`). `role="option"`,
     `aria-selected`.
  4. **"Remover tag"** (se `allowClear` e `value != null`): item que chama `onChange(null)`.
  5. **Estado vazio do filtro:** se a busca não casa nenhuma tag e `onCreate` existe, mostra só o "+ Criar
     «{query}»"; se não há `onCreate`, mostra `EmptyState` mínimo "Nenhuma tag encontrada".
- **Placeholder "Selecione uma tag":** é o texto do trigger quando `value == null` — atende o print.

#### Estados internos

- `open: boolean` — popover aberto.
- `query: string` — texto do filtro "Buscar tag…".
- `creating: boolean` — true enquanto a promise de `onCreate` resolve (desabilita o "+ Criar", mostra
  spinner inline). Em erro (`ApiError`, ex. 409 nome duplicado), exibe a mensagem inline e mantém o popover
  aberto.
- `activeIndex: number` — item destacado para navegação por teclado (↑/↓).

### 6.2 Hooks (TanStack Query)

```ts
// web/lib/hooks/useTags.ts
import { useQuery } from "@tanstack/react-query";
import { api } from "@/lib/api";
import type { Tag } from "@/lib/types";

export type TagWithCount = Tag & { tx_count: number };

export function useTags() {
  return useQuery({
    queryKey: ["tags"],
    queryFn: () => api.get<{ items: TagWithCount[] }>("/tags").then((r) => r.items),
    staleTime: 1000 * 60 * 10,   // tags mudam pouco
  });
}
```

```ts
// web/lib/hooks/useTagMutations.ts
import { useMutation, useQueryClient } from "@tanstack/react-query";
import { api } from "@/lib/api";
import type { Tag } from "@/lib/types";

export function useCreateTag() {
  const qc = useQueryClient();
  return useMutation({
    mutationFn: (vars: { name: string; color?: string | null }) =>
      api.post<Tag & { tx_count: number }>("/tags", vars),
    onSuccess: () => qc.invalidateQueries({ queryKey: ["tags"] }),
  });
}

export function useUpdateTag() {
  const qc = useQueryClient();
  return useMutation({
    mutationFn: (vars: { id: number; name?: string; color?: string | null }) =>
      api.patch<Tag & { tx_count: number }>(`/tags/${vars.id}`, {
        name: vars.name, color: vars.color,
      }),
    onSuccess: () => qc.invalidateQueries({ queryKey: ["tags"] }),
  });
}

export function useDeleteTag() {
  const qc = useQueryClient();
  return useMutation({
    mutationFn: (id: number) => api.del<{ deleted_id: number; untagged: number }>(`/tags/${id}`),
    onSuccess: () => {
      qc.invalidateQueries({ queryKey: ["tags"] });
      qc.invalidateQueries({ queryKey: ["transactions"] }); // linhas desvinculadas (F2.2)
    },
  });
}
```

```ts
// web/lib/hooks/useSetTag.ts  (análogo a useSetBucket de F1.1; usado por F2.2/F2.4)
import { useMutation, useQueryClient } from "@tanstack/react-query";
import { api } from "@/lib/api";

export function useSetTag() {
  const qc = useQueryClient();
  return useMutation({
    mutationFn: (vars: { txId: string; tagId: number | null }) =>
      api.patch(`/transactions/${vars.txId}`, { tag_id: vars.tagId }),
    onSuccess: () => qc.invalidateQueries({ queryKey: ["transactions"] }),
  });
}
```

> **`onCreate` do `TagSelect`** (usado inline em F2.2): o caller passa
> `onCreate={(name) => createTag.mutateAsync({ name })}` (de `useCreateTag`), que retorna a `Tag` criada;
> o `TagSelect` a seleciona via `onChange(tag.id)` → o caller persiste com `useSetTag`. Assim "criar +
> selecionar" é um fluxo de 2 mutations encadeadas, ambas invalidando os caches certos.

### 6.3 Criação on-the-fly (fluxo "+ Criar")

1. Usuário abre o `TagSelect`, digita "Pets" no "Buscar tag…". Nenhuma tag casa exatamente.
2. O topo mostra **"+ Criar «Pets»"**. Usuário clica.
3. `creating=true`; chama `onCreate("Pets")` → `POST /api/tags { name: "Pets" }` (cor: ausente → backend
   grava `NULL`; a UI usa cor neutra padrão até o usuário recolorir na página `/tags`).
   - **Sucesso (201):** recebe `{ id, name, color, tx_count:0 }`; o cache `["tags"]` é invalidado
     (`useCreateTag.onSuccess`); o `TagSelect` chama `onChange(novaTag.id)` (seleciona) e fecha.
   - **Conflito (409):** se "Pets" já existia (ex.: corrida), `ApiError(code:'conflict')`; a UI mostra
     inline _"já existe uma tag com esse nome"_ e — recomendado — **seleciona a tag existente** (busca em
     `options` por nome case-insensitive) em vez de falhar bruscamente.
   - **Validação (422):** nome inválido → mensagem inline; popover permanece aberto.
4. **Cor na criação on-the-fly:** o seletor cria **sem cor** (rápido, 1 clique). A escolha de cor fica para
   a página `/tags` (modal completo) ou para o `PATCH` posterior. *(Alternativa considerada: pedir cor já no
   dropdown — preterida por fricção; o print mostra "+ Criar" como ação simples no topo.)*

  `#F5B301` (amarelo/accent), `#EF4444` (vermelho), `#38BDF8` (azul-claro), `#9F1239` (vinho), `#3B82F6`

  Abaixo, um `<input type="text">` para hex livre (validado no front com o mesmo regex `^#([0-9a-f]{3}|[0-9a-f]{6})$`
  case-insensitive antes de enviar; o backend revalida).
- **Ações do modal:** **Salvar** (cria via `useCreateTag` ou edita via `useUpdateTag`) e **Cancelar**.
- **Erros:** `409` (nome duplicado) e `422` (cor/nome inválido) exibidos inline no modal (de `ApiError.message`).
- **Edição inline (alternativa rápida):** o print mostra "editar" como ação na linha; esta spec implementa
  edição **via o mesmo modal preenchido** (mais simples e consistente que edição inline na célula). *(Edição
  inline célula-a-célula fica como melhoria opcional; o requisito "editar inline/excluir" é atendido pelo
  modal de edição acionado na linha.)*

**Exclusão (confirmação):**
- Clicar na lixeira abre uma **confirmação** (modal/dialog): _"Excluir a tag «{nome}»? N transações ficarão
  sem tag (as transações não são apagadas)."_ — usando `tx_count` para informar o impacto.
- Confirmar → `useDeleteTag.mutate(id)` → `DELETE /api/tags/{id}`. Sucesso: invalida `["tags"]` e
  `["transactions"]`; toast _"Tag excluída. N transações desvinculadas."_ (de `untagged` da resposta).
- A confirmação é **só de UI** (o backend exclui direto — §4.4).

**Estados:**
- **Loading:** `useTags().isLoading` → `DataTable loading` (skeleton).
- **Vazio:** `data.length === 0` → `EmptyState` _"Nenhuma tag ainda"_ + CTA "+ Criar Tag" (raro em operação
  normal por causa do seed, mas tratado).
- **Erro:** `useTags().isError` → bloco de erro com `ApiError.message` + botão "Tentar novamente"
  (`refetch`).

### 6.5 Acessibilidade

- `TagSelect`: trigger é `<button>` com `aria-haspopup="listbox"`/`aria-expanded`; popover `role="listbox"`;
  itens `role="option"` com `aria-selected`; navegação ↑/↓/Enter/Esc; foco volta ao trigger ao fechar;
  input de busca com `aria-label="Buscar tag"`. Bolinhas de cor são decorativas (`aria-hidden`) — a
  informação é o **nome** (cor não é o único portador de significado).
- Página `/tags`: tabela semântica (`<table>` via `DataTable`); botões de ação com `aria-label`
  ("Editar tag «{nome}»", "Excluir tag «{nome}»"); modal com foco preso (focus trap), `Esc` fecha, `aria-modal`.
- Contraste: nomes em `text-text`; as cores das tags são fundo de pill com texto contrastante (o `Pill` de
  F0.2 usa `color` no texto sobre `${color}22` de fundo — contraste suficiente para rótulos curtos).

## 7. Regras de negócio / edge cases

- **Nome único case-insensitive.** "Lazer" e "lazer" não coexistem. Validado por `name_taken` (`LOWER(name)`)
  **antes** do insert/update; o `UNIQUE` do banco (case-sensitive) é a rede de segurança. Duplicado → **409**.
  *(Decisão: case-insensitive porque o usuário não distingue "Saúde"/"saúde" visualmente; evita confusão.)*
- **Cor inválida.** Hex `#RGB`/`#RRGGBB` apenas; qualquer outra coisa (nome CSS "red", rgb(), hex sem `#`,
  4/8 dígitos) → **422**. Cor é **opcional**: ausente/`""`/`null` → `color=NULL`; a UI usa neutro `#9A9AA2`.
- **Excluir tag em uso.** Permitido. `delete_tag` desvincula (`tag_id=NULL`) **antes** de apagar a tag, na
  mesma transação; **nenhuma transação é apagada**; retorna `untagged`. A UI confirma e informa o impacto.
- **Criar on-the-fly.** `TagSelect` "+ Criar «query»" cria via `POST /api/tags` (sem cor) e já seleciona.
  Corrida (409) → seleciona a existente. Nome vazio após trim → o "+ Criar" fica desabilitado.
- **`tag_id` inválido no PATCH de transação.** `tag_exists=False` → **422** (`"tag_id inválido"`).
  `tag_id: null` limpa (sempre válido).
- **Limite de tags.** Sem limite rígido no backend (base pessoal). A UI lida bem com dezenas; se um dia
  passar de ~centenas, o `TagSelect` ganharia virtualização (registrado como melhoria, fora do escopo). O
  contador "N tags encontradas" não tem teto.
- **Renomear para o próprio nome.** `PATCH {name:"Saúde"}` numa tag já chamada "Saúde" **não** é 409 —
  `name_taken(..., exclude_id=tag_id)` ignora a própria linha. Salvar sem mudar o nome é permitido.
- **Tag-semente excluída + reboot.** O seed recria por `name` (§4.1, nota). Aceitável; documentado.
- **`tx_count` e performance.** O `LEFT JOIN ... GROUP BY` em `GET /api/tags` é O(transações) por
  agregação, mas a base é pessoal (10³–10⁴ linhas) e há índice `idx_tx_category`; um índice em
  `transactions(tag_id)` **pode** ser adicionado (não obrigatório nesta spec) se a agregação pesar. Mitigação
  registrada: índice `idx_tx_tag_id ON transactions(tag_id)` em migration futura (F2.2, junto dos filtros).
- **Concorrência.** `Database` é 1 conexão por request (F0.1/primer §2.1, `check_same_thread=False`); cada
  operação de tag é uma transação SQLite (`db._cursor()` commit/rollback). Sem multiusuário (primer §5.5),
  risco de corrida desprezível; o `409` cobre o raro double-submit.
- **Multi-tag (fora de escopo).** Modelo é 1 tag/transação (`tag_id` escalar). Se o produto pedir várias
  tags por transação, vira tabela de junção `transaction_tags(transaction_id, tag_id)` em spec própria — não
  nesta.
- **Auto-tag por regra (fora de escopo).** Diferente da Meta (F1.1), tag não tem `bucket_rules`/propagação.
  Setar tag afeta só a transação. Auto-sugestão de tag por comerciante é melhoria futura.

## 9. Critérios de aceite (verificáveis)

1. **Seed:** num banco novo, após subir o app, `GET /api/tags` retorna **exatamente 7** tags com os nomes
   `Alimentação, Conforto, Educação, Lazer, Saúde, Transporte, Vestuário`, cada uma com `color` hex e
   `tx_count` (0 em base nova). Rodar o seed de novo **não** cria duplicatas e **não** reverte uma `color`
   editada.
2. **Listagem com contador:** `GET /api/tags` inclui `tx_count` por tag (via `LEFT JOIN`); tags sem uso vêm
   com `tx_count: 0`; a lista é ordenada por `name` case-insensitive; envelope `{ items: [...] }`.
3. **Criar:** `POST /api/tags { "name": "Pets", "color": "#10B981" }` → **201** com
   `{ id, name:"Pets", color:"#10b981", tx_count:0 }`. Sem `color` → cria com `color:null`.
4. **Nome único (409):** `POST /api/tags { "name": "saúde" }` quando já existe "Saúde" → **409 conflict**
   (`error.code == "conflict"`). Idem ao renomear via `PATCH` para um nome de outra tag.
5. **Cor inválida (422):** `POST`/`PATCH` com `color: "vermelho"` ou `"#12"` ou `"ff0000"` (sem `#`) →
   **422 validation_error**.
6. **Editar:** `PATCH /api/tags/{id} { "name": "Saúde e Bem-estar" }` muda só o nome; `{ "color": "#22C55E" }`
   muda só a cor; `{ "color": null }` limpa a cor; id inexistente → **404 not_found**.
7. **Excluir desvincula (não apaga):** dado N transações com `tag_id = X`, `DELETE /api/tags/X` → **200**
   `{ "deleted_id": X, "untagged": N }`; após isso, **as N transações ainda existem** (count inalterado) com
   `tag_id IS NULL`; a tag X não existe mais. Id inexistente → **404**.
8. **`tag_id` em transação:** `PATCH /api/transactions/{id} { "tag_id": 3 }` grava `tag_id=3`;
   `{ "tag_id": null }` limpa; `tag_id` inexistente → **422** (`"tag_id inválido"`); id de transação
   inexistente → **404**. **Sem** propagação (afeta só a transação).
9. **`TagSelect`:** placeholder "Selecione uma tag"; popover com "Buscar tag…", "+ Criar «query»" no topo,
   lista de pills coloridas; selecionar dispara `onChange(id)`; "+ Criar" cria on-the-fly (`POST /api/tags`),
   seleciona a nova tag e fecha; "Remover tag" chama `onChange(null)`.
10. **Página `/tags`:** cabeçalho com o texto do print, contador "**N tags encontradas**" correto, tabela
    com bolinha de cor + nome + ações editar/excluir, botão amarelo "+ Criar Tag" que abre o modal (nome +
    paleta + hex). Excluir mostra confirmação com o impacto (`tx_count`) e, após sucesso, a lista atualiza e
    aparece o toast com `untagged`. Estados loading/vazio/erro presentes.
11. **Consistência com F1.1:** `GET /api/tags` usa o mesmo envelope `{items}` de `GET /api/buckets`; o
    `TagSelect` tem UX/props análogas ao `BucketSelect`; o `PATCH /api/transactions/{id}` aceita `tag_id` e
    `bucket_id` sem conflito.
12. **Não-regressão:** front legado (`src/web/`) intacto; `/api/summary\|dashboard\|cartao` inalterados;
    `budget.py`/`categorizer.py` intactos; testes existentes verdes.

## 11. Notas de decisão (resumo)

- **`tx_count` incluído no `GET /api/tags`** (via `LEFT JOIN`): o primer o marca como opcional; incluí-lo
  habilita o contexto de exclusão na UI e tem custo desprezível na escala pessoal. Índice
  `transactions(tag_id)` fica como mitigação futura (F2.2).
- **Exclusão desvincula em vez de bloquear/cascatear:** `DELETE` faz `UPDATE transactions SET tag_id=NULL`
  **antes** de apagar a tag, na mesma transação SQLite (atômico), e retorna `untagged`. Nenhuma transação é
  perdida — coerente com o requisito central e com a FK ser apenas **lógica** (F0.1 §3.4: sem `ON DELETE`).
- **Nome único case-insensitive na aplicação** (com `UNIQUE` do banco como rede de segurança): evita
  "Saúde"/"saúde" duplicados que confundiriam o usuário; duplicidade → **409**, validação de forma → **422**.
- **Tag não propaga (diferente da Meta de F1.1):** setar `tag_id` afeta só a transação; **não** há
  `bucket_rules`/`POST /{id}/tag`. Isso mantém o modelo simples e reflete que tag é escolha pontual.
- **`TagSelect` cria on-the-fly sem cor (1 clique):** o "+ Criar «query»" no topo do dropdown reflete o
  print e prioriza fluidez na edição inline (F2.2); a coloração fina é feita na página `/tags`. Em corrida
  (409), seleciona a tag existente em vez de falhar.
